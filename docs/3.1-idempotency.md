# 3.1 — Idempotency

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích idempotency — "gửi cùng request N lần → kết quả giống lần đầu"
- Tôi có thể giải thích defense-in-depth: Redis layer (fast) + DB layer (durable)
- Tôi có thể giải thích tại sao idempotency key phải client-generated, không server-generated
- Tôi có thể giải thích race condition khi 2 duplicate requests đến đồng thời
- Tôi có thể giải thích TTL strategy: quá ngắn → duplicate không bị bắt, quá dài → Redis memory

## The Mental Model

Hình dung **ATM rút tiền**:
1. Bạn nhấn "Rút 500k" → màn hình freeze → bạn nhấn lại → nhấn lại → nhấn 5 lần
2. **Không idempotent**: rút 5 lần = mất 2.5M 💀
3. **Idempotent**: ATM nhận ra 5 lần nhấn cùng 1 giao dịch → rút 1 lần = 500k ✅

**Cách thực hiện**: mỗi lần nhấn, ATM gửi cùng `transaction_id`. Server kiểm tra:
- `transaction_id` chưa thấy → xử lý + lưu kết quả
- `transaction_id` đã thấy → trả kết quả cũ, KHÔNG xử lý lại

## Tại sao cần idempotency cho ticketing

```
Scenario: User mua vé, mạng chập chờn

T=0ms   Client: POST /tickets/reserve-and-buy {Idempotency-Key: "abc-123"}
T=100ms Server: đang xử lý... seat locked, ticket creating...
T=5000ms Client: timeout → retry → POST /tickets/reserve-and-buy {Idempotency-Key: "abc-123"}
T=5050ms Server: xử lý request thứ 2 → TẠO THÊM 1 TICKET → double-charge user!

Với idempotency:
T=5050ms Server: Idempotency-Key "abc-123" đã thấy trong Redis
         → Nếu đang xử lý: return 409 "Still processing"
         → Nếu đã xong: return cached response (same 201 + ticketId)
         → KHÔNG tạo ticket thứ 2 ✅
```

## Cơ chế hoạt động — Defense in Depth

### Layer 1: Redis (fast, in-memory) — IdempotencyFilter

```
Request flow qua IdempotencyFilter:

1. Extract Idempotency-Key header
2. Redis SET NX "idempotency:abc-123" "__PROCESSING__" EX 3600
   → SET NX = Set-If-Not-Exists (atomic)
   → Trả TRUE nếu key mới → request ĐẦU TIÊN → tiếp tục xử lý
   → Trả FALSE nếu key đã tồn tại → DUPLICATE:
     a) Value = "__PROCESSING__" → request gốc đang chạy → return 409
     b) Value = cached response → return cached (same status + body)

3. Sau khi xử lý xong:
   Redis SET "idempotency:abc-123" "{status:201, body:{ticketId:42}}" EX 3600
   → Cache TOÀN BỘ response (status code + body)

4. Nếu xử lý FAIL:
   Redis DEL "idempotency:abc-123"
   → Xóa key → cho phép client retry
```

### Layer 2: Database (durable) — idempotency_key on tickets

```
Ticket table có column: idempotency_key VARCHAR UNIQUE

Khi tạo ticket:
  INSERT INTO tickets (event_id, seat_id, user_id, idempotency_key)
  VALUES (1, 42, 'user-abc', 'abc-123')

Nếu duplicate đến sau Redis TTL hết:
  → Redis không còn key → request đi qua filter
  → Strategy kiểm tra: SELECT * FROM tickets WHERE idempotency_key = 'abc-123'
  → Tìm thấy → return existing ticket (không tạo mới)
  → UNIQUE constraint là safety net cuối cùng
```

### Tại sao cần CẢ HAI layers

```
Chỉ Redis:
  ✅ Fast (0.1ms)
  ❌ Volatile — Redis restart → mất tất cả idempotency keys
  ❌ TTL hết → duplicate không bị bắt

Chỉ DB:
  ✅ Durable — survive restart
  ❌ Slow (~5ms per lookup)
  ❌ Không bắt được duplicate TRONG KHI đang xử lý (race condition)

Cả hai:
  Redis: bắt duplicate nhanh (99% cases), xử lý PROCESSING state
  DB: bắt duplicate sau TTL hết (1% edge case), UNIQUE constraint
  → Defense in depth: không layer nào bỏ sót
```

## Race Condition Analysis

### Concurrent duplicate requests

```
T=0ms  Request A: SET NX "abc-123" → TRUE (claimed)
T=0ms  Request B: SET NX "abc-123" → FALSE (blocked)
       → Request B thấy PROCESSING → return 409

Không race condition ở Redis level — SET NX là atomic.
```

### TTL expiry during processing

```
T=0     Request A: SET NX → TRUE, bắt đầu xử lý
T=3600s TTL hết: Redis xóa key
T=3601s Request B: SET NX → TRUE (key không còn → B nghĩ mình là đầu tiên!)
        → B tạo ticket → duplicate!

Giải pháp: Layer 2 (DB)
  → B: findByIdempotencyKey("abc-123") → tìm thấy ticket A đã tạo
  → B: return existing ticket, KHÔNG tạo mới
```

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Ngăn double-charge hoàn toàn | Thêm Redis dependency cho idempotency |
| Defense in depth (Redis + DB) | TTL cần tuning: 1h default |
| Transparent — client chỉ cần thêm header | Client phải generate idempotency key |
| Backward compatible — không có header = xử lý bình thường | Redis memory cho cached responses |
| O(1) duplicate check | Cached response chỉ cho POST, không cho GET |

## Real-World Context

### Stripe API — Idempotency-Key header
- Stripe yêu cầu `Idempotency-Key` cho mọi POST request
- Key do client tạo (UUID v4)
- TTL = 24h — sau 24h key bị xóa
- Response cached bao gồm cả error responses

### AWS S3 — Natural idempotency
- PUT object với cùng key → overwrite (idempotent by design)
- Không cần idempotency key — operation tự nhiên idempotent

### gRPC — At-least-once delivery
- gRPC mặc định retry on failure → cần idempotency ở server
- Idempotency key thường là request metadata

## Khi nào dùng trong production

✅ Luôn dùng cho:
- **Payment/billing operations** — double-charge = lawsuit
- **Ticket/order creation** — double-book = revenue loss + angry customer
- **Any POST that creates a resource** — duplicate = data inconsistency

❌ Không cần cho:
- **GET requests** — naturally idempotent
- **DELETE requests** — thường idempotent (delete đã xóa = no-op)
- **PUT requests** — thường idempotent (overwrite)

### Idempotency key flow through system

```

  → IdempotencyFilter: Redis SET NX (claim)
    → TicketController: read header, pass to strategy
      → Strategy: check DB for existing ticket with key
        → If found: return existing
        → If not: create ticket with idempotency_key = "abc-123"
    → IdempotencyFilter: cache response in Redis
```
