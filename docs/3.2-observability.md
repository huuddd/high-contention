# 3.2 — Observability

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích tại sao observability là bắt buộc cho hệ thống high contention
- Tôi có thể giải thích Micrometer counters hoạt động thế nào (atomic CAS, O(1), lock-free)
- Tôi có thể giải thích cách expose metrics qua Actuator + Prometheus endpoint
- Tôi có thể giải thích `/events/{id}/stats` endpoint và cách nó aggregate dữ liệu real-time

## The Mental Model

Hình dung **bảng điều khiển nhà máy**:
- Bạn có 6 dây chuyền sản xuất (6 strategy) cùng sản xuất 1 sản phẩm (ticket)
- Mỗi dây chuyền có đồng hồ đo: sản phẩm OK, lỗi, retry, deadlock, timeout
- Không có bảng điều khiển → bạn chỉ biết "hỏng" nhưng không biết **tại sao** và **bao nhiêu**
- Có bảng điều khiển → so sánh 6 dây chuyền bằng **số liệu**, không bằng cảm tính

## Tại sao cần observability

```
Scenario: Benchmark 200 concurrent users, Strategy B (Pessimistic)
Kết quả: 95% success, 5% failure

Không có metrics:
  → "5% lỗi" — do deadlock? timeout? oversell? → KHÔNG BIẾT

Có metrics:
  → ticketing.conflicts{strategy=pessimistic} = 0
  → ticketing.deadlocks{strategy=pessimistic} = 8
  → ticketing.timeouts{strategy=pessimistic} = 2
  → ticketing.oversells{strategy=pessimistic} = 0
  → Kết luận: 5% failure = 8 deadlocks + 2 timeouts, KHÔNG có oversell ✅
```

## Cơ chế hoạt động

### Micrometer Counters

```
ConflictMetrics sử dụng Micrometer Counter:
  - Counter là atomic (CAS operation) — thread-safe, lock-free
  - Overhead: <1μs per increment — negligible so với DB operation (~5ms)
  - Tag "strategy" cho phép filter metrics theo strategy đang active

Các counters:
  ticketing.conflicts    — OCC version mismatch, serialization failure
  ticketing.retries      — retry attempts (OCC, SERIALIZABLE)
  ticketing.deadlocks    — PostgreSQL 40P01 deadlock detected
  ticketing.timeouts     — lock wait timeout, connection timeout
  ticketing.oversells    — available_seats < 0 (should NEVER happen)
  ticketing.successes    — successful ticket purchase
  ticketing.failures     — failed after max retries or business rule
```

### Stats Endpoint — `/events/{id}/stats`

```
GET /events/1/stats

Response:
{
  "eventId": 1,
  "eventName": "Concert ABC",
  "totalSeats": 100,
  "availableSeats": 42,
  "soldCount": 58,
  "consistent": true,
  "strategy": "pessimistic",
  "metrics": {
    "conflicts": 12,
    "retries": 15,
    "deadlocks": 3,
    "timeouts": 1,
    "oversells": 0,
    "successes": 58,
    "failures": 5
  }
}

→ consistent = (availableSeats + soldCount == totalSeats)
→ metrics từ Micrometer counters, filtered by active strategy
```

### Prometheus Endpoint

```
GET /actuator/prometheus

# HELP ticketing_conflicts_total Number of concurrency conflicts detected
# TYPE ticketing_conflicts_total counter
ticketing_conflicts_total{strategy="occ",application="ticketing-service"} 42.0

# HELP ticketing_successes_total Number of successful ticket purchases
# TYPE ticketing_successes_total counter
ticketing_successes_total{strategy="occ",application="ticketing-service"} 958.0
```

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Real-time visibility vào conflict patterns | Thêm dependency Micrometer (đã có sẵn qua Actuator) |
| So sánh strategies bằng số liệu cụ thể | Counter chỉ tăng, không giảm — cần reset giữa benchmark runs |
| O(1) overhead per metric increment | Stats endpoint thêm DB query (countByEventId) |
| Prometheus-compatible → Grafana dashboard | Metrics reset khi restart application |

## Implementation

### ConflictMetrics (đã tạo trong 1.4)

7 counters: conflicts, retries, deadlocks, timeouts, oversells, successes, failures.
Mỗi counter tagged với `strategy` name.

### TicketingStatsService (mới)

Aggregate metrics từ Micrometer registry:
- Query counter values by name + strategy tag
- Combine với event data từ DB
- Return unified stats response

### Enhanced EventController `/events/{id}/stats`

Thêm Micrometer metrics vào stats response:
- Read counter values từ MeterRegistry
- Include active strategy name
- Consistency check: available + sold = total

## Real-World Context

### Netflix — Resilience metrics
- Hystrix dashboard: real-time circuit breaker metrics
- Counters: success, failure, timeout, short-circuit, rejected
- Same pattern: atomic counters → aggregate → expose

### Stripe — API metrics
- Per-endpoint: request_count, error_rate, latency_p50/p95/p99
- Tagged by: endpoint, method, status_code
- Alert on: error_rate > 1%, latency_p99 > 5s

## Khi nào dùng trong production

✅ Luôn dùng:
- **Benchmark comparison** — so sánh strategies bằng số liệu
- **Production monitoring** — detect oversell, deadlock spikes
- **Capacity planning** — retry rate tăng = cần scale hoặc đổi strategy

❌ Cẩn thận:
- **High cardinality tags** — đừng tag theo user_id (quá nhiều unique values)
- **Counter reset** — restart app = mất metrics → cần Prometheus scraping liên tục
