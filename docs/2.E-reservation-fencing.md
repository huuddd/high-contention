# 2.E — Reservation + TTL + Fencing Token

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích tại sao tách "giữ chỗ" (reserve) khỏi "thanh toán" (confirm) là thiết kế UX-oriented
- Tôi có thể giải thích TTL (Time-To-Live) — tại sao reservation phải có thời hạn, hậu quả nếu không
- Tôi có thể giải thích Fencing Token — tại sao UUID token ngăn stale confirmation
- Tôi có thể giải thích background cleanup job — tại sao cần, race condition khi expire
- Tôi có thể so sánh Reservation vs Pessimistic/OCC: contention window khác nhau như thế nào

## The Mental Model

Hình dung **đặt bàn nhà hàng**:
1. **Reserve**: "Giữ bàn 15 cho tôi" → nhà hàng đánh dấu bàn 15 là "ĐÃ GIỮ", ghi tên bạn + hẹn 10 phút
2. **Confirm**: Bạn đến trong 10 phút, đưa "phiếu giữ bàn" → nhà hàng xác nhận, bàn là của bạn
3. **Expire**: 10 phút mà không đến → nhà hàng xóa tên, bàn trở lại "TRỐNG" cho người khác
4. **Fencing Token**: Phiếu giữ bàn có số unique → nếu ai đó photocopy phiếu, chỉ phiếu gốc có hiệu lực

**Tại sao cần tách reserve/confirm?**
- Strategies A-D: lock/check-then-update trong **1 transaction** → contention window = toàn bộ operation
- Reservation: lock chỉ trong **reserve** (rất nhanh) → unlock → user có thời gian thanh toán → **confirm** (cũng nhanh)
- Contention window: A-D = cả flow; E = chỉ reserve step

```
Strategies A-D:
  [--- LOCK --- CHECK --- UPDATE --- INSERT --- COMMIT ---]
  ←──────────── contention window (toàn bộ) ──────────────→

Strategy E:
  [--- RESERVE (fast, lock) ---]  ...user pays...  [--- CONFIRM (fast, lock) ---]
  ←── contention ──→               ←── free ──→     ←── contention ──→
```

## Tại sao strategies trước chưa đủ

### Pessimistic (2.B) — lock quá lâu
```
User flow thực tế:
  1. Chọn ghế (100ms)
  2. Nhập thông tin thanh toán (30s — user typing)
  3. Xác nhận thanh toán (2s — payment gateway)
  4. Tạo ticket (5ms)

Pessimistic lock toàn bộ: 30s + 2s + ... = giữ lock >30s
→ 200 concurrent users × 30s/user = 100 phút queue
→ User timeout, connection pool exhaustion, 💀
```

### OCC/SERIALIZABLE (2.C/2.D) — wasted work
```
User chọn ghế → bắt đầu thanh toán → 30s sau → UPDATE
→ Version đã thay đổi 50 lần trong 30s → retry storm
→ Mỗi retry = gọi lại payment gateway = 💰💰💰
```

### Reservation — tách contention window
```
Reserve (5ms, pessimistic lock):
  → Lock seat, decrement counter, tạo reservation + fencing token
  → Release lock

User thanh toán (30s, KHÔNG lock):
  → Seat status = LOCKED, không ai khác có thể mua
  → Nhưng KHÔNG giữ DB lock → connection pool free

Confirm (5ms, pessimistic lock):
  → Verify fencing token + expiry
  → Tạo ticket, mark seat SOLD
  → Release lock

Expire (background, mỗi 30s):
  → Tìm reservations quá hạn → release seats
```

## Cơ chế hoạt động (DB/OS level)

### Fencing Token — tại sao cần

```
Scenario KHÔNG có fencing token:

T=0s    User A: POST /reservations → reservation_id=42, seat locked
T=5s    User A: mất mạng, không confirm
T=600s  Expiry job: reservation 42 expired → seat released
T=601s  User B: POST /reservations → reservation_id=43, seat locked
T=602s  User A: mạng trở lại, POST /reservations/42/confirm
        → ❌ STALE confirmation! Seat đã thuộc reservation 43 của User B

Giải pháp: Fencing Token (UUID)
  Reserve: tạo token = UUID, lưu trong reservation
  Confirm: kiểm tra token match + reservation chưa expired
  → User A confirm với token cũ → token mismatch hoặc status != PENDING → REJECT
```

### TTL (Time-To-Live) — tại sao cần background cleanup

```
Nếu KHÔNG có cleanup:
  → User reserve 100 seats, không bao giờ confirm
  → 100 seats bị LOCKED mãi mãi → "ghost reservations"
  → available_seats = 0 nhưng không ai mua được → lost revenue

Background job (mỗi 30s):
  UPDATE reservations SET status = 'EXPIRED'
  WHERE status = 'PENDING' AND expires_at < NOW();
  
  UPDATE seats SET status = 'AVAILABLE', locked_by = NULL, locked_until = NULL
  WHERE id IN (expired_reservation_seat_ids);
  
  UPDATE events SET available_seats = available_seats + count(expired)
  WHERE id = event_id;
```

### Race condition: confirm vs expire

```
T=599.9s  User A: POST /confirm (token=abc, reservation still PENDING)
T=600.0s  Expiry job: finds reservation PENDING + expired → marks EXPIRED

Ai thắng?

Giải pháp: atomic status transition
  UPDATE reservations SET status = 'CONFIRMED'
  WHERE id = 42 AND status = 'PENDING' AND fencing_token = :token AND expires_at > NOW()
  
  affected_rows = 1 → confirm thắng (expiry job sẽ skip vì status != PENDING)
  affected_rows = 0 → expired hoặc token sai → reject confirm
  
→ Status transition là atomic → không race condition
```

### Seat state machine

```
AVAILABLE ──reserve──→ LOCKED ──confirm──→ SOLD
    ↑                     │
    └────expire/cancel────┘
```

## SQL / Implementation

### Reserve flow
```sql
BEGIN;
-- Lock event + seat (pessimistic, ngắn)
SELECT * FROM events WHERE id = :eventId FOR UPDATE;
SELECT * FROM seats WHERE event_id = :eventId AND seat_label = :seatLabel FOR UPDATE;

-- Check availability
-- Decrement counter
UPDATE events SET available_seats = available_seats - 1 WHERE id = :eventId;

-- Lock seat
UPDATE seats SET status = 'LOCKED', locked_by = :userId,
  locked_until = NOW() + INTERVAL '10 minutes'
WHERE id = :seatId AND status = 'AVAILABLE';

-- Create reservation with fencing token
INSERT INTO reservations (event_id, seat_id, user_id, fencing_token, status, expires_at)
VALUES (:eventId, :seatId, :userId, :fencingToken, 'PENDING', NOW() + INTERVAL '10 minutes');

COMMIT;
-- Lock released — user có 10 phút để confirm
```

### Confirm flow
```sql
BEGIN;
-- Atomic status transition — fencing token + expiry check
UPDATE reservations SET status = 'CONFIRMED'
WHERE id = :reservationId AND fencing_token = :token
  AND status = 'PENDING' AND expires_at > NOW();
-- affected_rows = 0 → reject (expired hoặc wrong token)

-- Mark seat as SOLD
UPDATE seats SET status = 'SOLD' WHERE id = :seatId AND status = 'LOCKED';

-- Create ticket
INSERT INTO tickets (event_id, seat_id, user_id) VALUES (:eventId, :seatId, :userId);

COMMIT;
```

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Contention window cực ngắn (reserve + confirm = ~10ms) | 2-step UX — phức tạp hơn 1-click buy |
| User có thời gian thanh toán mà không block DB | Ghost reservations nếu cleanup job chậm |
| Fencing token ngăn stale confirmation | Thêm background job — complexity |
| Seat LOCKED visible trên UI (UX tốt) | TTL tuning: quá ngắn → user bực; quá dài → seat bị hold lâu |
| Scale horizontally — không giữ DB lock dài | 2 HTTP calls thay vì 1 — network overhead |

## Latency / Throughput Analysis

```
Reserve: ~5ms (pessimistic lock, short)
Confirm: ~5ms (atomic update, short)
Total contention per user: ~10ms (vs 30s+ cho Pessimistic full-flow)

Throughput:
  Pessimistic (full flow 30s): 1000ms / 30000ms ≈ 33 TPS
  Reservation (reserve 5ms):  1000ms / 5ms = 200 TPS just for reserve
  
  200 users concurrent:
    Pessimistic p95: 190 × 30s = 95 MINUTES 💀
    Reservation reserve p95: 190 × 5ms = 950ms ✅
    Reservation confirm p95: 190 × 5ms = 950ms ✅
```

## Real-World Context

### Ticketmaster / StubHub
- Reserve step: "Hold your seats for 10 minutes"
- Confirm step: "Complete payment to secure your tickets"
- Expire: countdown timer on checkout page — seats released if not paid

### Amazon Cart → Checkout
- Add to cart ≠ reserved (soft hold)
- "Proceed to checkout" = reservation with ~15 min TTL
- Payment = confirm
- Timeout = items back to inventory

### Booking.com
- "Book now, pay later" = reservation with 24h TTL
- No fencing token needed (payment is the confirmation, not a separate step)

## Khi nào dùng trong production

✅ Phù hợp:
- **Multi-step checkout** — user cần thời gian thanh toán
- **High concurrency + long user interaction** — giảm contention window 100x
- **UX requirement** — "Your seat is held for 10 minutes"
- **Payment integration** — tách payment khỏi DB lock

❌ Không phù hợp:
- **1-click buy** — overhead không cần thiết, dùng Pessimistic/OCC
- **Real-time bidding** — TTL quá chậm, cần instant resolution
- **Low contention** — thêm complexity không đáng

→ Reservation là **production pattern #1** cho ticketing, e-commerce, booking
