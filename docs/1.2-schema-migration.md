# 1.2 — Flyway Schema Migration

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích tại sao schema design quyết định hiệu quả của concurrency strategies
- Tôi có thể giải thích CHECK constraint hoạt động ở PostgreSQL level — khi nào nó fire, có atomic không
- Tôi có thể giải thích tại sao `version` column cần thiết cho OCC (2.C)
- Tôi có thể giải thích UNIQUE constraint là lightweight concurrency guard — nó prevent double-book ở DB level
- Tôi có thể giải thích tại sao Flyway migrations > Hibernate ddl-auto trong production

## The Mental Model

Hình dung database schema như **bản thiết kế phòng hát nhạc** trước khi bán vé:
- `events` = buổi hòa nhạc — có tổng số ghế, đếm ghế còn trống
- `seats` = từng ghế cụ thể — có nhãn (A1, A2...), trạng thái (AVAILABLE/LOCKED/SOLD)
- `tickets` = vé đã bán — liên kết event + seat + user, là **bằng chứng quyền sở hữu**
- `reservations` = giữ chỗ tạm — có thời hạn (TTL), có fencing token để chống stale confirm

Mối quan hệ: Event 1—N Seats, Seat 1—0..1 Ticket, Seat 1—0..1 Reservation (active)

## Tại sao schema design quan trọng cho concurrency

Schema **không chỉ là nơi lưu trữ** — nó là **tuyến phòng thủ đầu tiên** chống vi phạm invariants:

### Invariant → Schema Guard

| Invariant | Guard tại schema level |
|-----------|----------------------|
| available_seats ≥ 0 | `CHECK (available_seats >= 0)` trên `events` |
| Không double-book | `UNIQUE (event_id, seat_id)` trên `tickets` |
| Không oversell | `CHECK` + application logic (strategy-dependent) |
| Fencing token validity | `expires_at` + `status` trên `reservations` |

### CHECK constraint — hoạt động như thế nào ở PG level?

```
BEGIN;
UPDATE events SET available_seats = available_seats - 1 WHERE id = 1;
-- PostgreSQL kiểm tra CHECK (available_seats >= 0) NGAY tại thời điểm UPDATE
-- Nếu vi phạm → raise exception, transaction abort
-- Điều này xảy ra TRƯỚC khi COMMIT — không cần application check
COMMIT;
```

**Quan trọng:** CHECK constraint evaluate **per-row, per-statement** — không phải per-transaction.
Nghĩa là nếu 2 concurrent transactions cùng `UPDATE available_seats = available_seats - 1`:
- Với READ COMMITTED: cả 2 có thể thành công nếu available_seats ban đầu >= 2
- CHECK constraint **không thay thế** locking/OCC — nó là **safety net cuối cùng**

### UNIQUE constraint — lightweight concurrency guard

```sql
UNIQUE (event_id, seat_id) ON tickets
```

Khi 2 transactions cùng INSERT ticket cho cùng seat:
- Transaction đầu tiên INSERT → thành công
- Transaction thứ hai INSERT → `duplicate key value violates unique constraint`
- PostgreSQL dùng **unique index lock** (khác với row lock) — short-lived, chỉ hold trong INSERT

→ UNIQUE constraint là **last line of defense** chống double-book, kể cả khi application logic có bug

## Cơ chế hoạt động — Flyway

### Tại sao Flyway, không phải Hibernate ddl-auto?

| | Flyway | Hibernate ddl-auto |
|--|--------|-------------------|
| Reproducibility | ✅ Cùng SQL chạy trên mọi môi trường | ❌ Generate SQL khác nhau tùy config |
| Version control | ✅ Migration files trong Git | ❌ Không track được lịch sử |
| Production-safe | ✅ Incremental, rollback-aware | ❌ `create-drop` = mất data |
| Complex changes | ✅ Custom SQL, data migration | ❌ Chỉ schema changes đơn giản |
| Team workflow | ✅ Conflict detection qua version number | ❌ Merge conflicts khó detect |

### Flyway execution flow

```
App starts → Flyway checks `flyway_schema_history` table
  → Nếu V1 chưa chạy → execute V1__init_schema.sql
  → Ghi record vào `flyway_schema_history` (version, checksum, timestamp)
  → Nếu V1 đã chạy → checksum match? → skip
                                       → checksum mismatch? → FAIL (migration tampered)
```

## SQL Implementation

### Table: `events`

```sql
CREATE TABLE events (
    id              BIGSERIAL PRIMARY KEY,
    name            VARCHAR(255) NOT NULL,
    total_seats     INT NOT NULL,
    available_seats INT NOT NULL,
    version         BIGINT NOT NULL DEFAULT 0,  -- OCC version counter (2.C)
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_available_seats_non_negative CHECK (available_seats >= 0),
    CONSTRAINT chk_available_lte_total CHECK (available_seats <= total_seats)
);
```

- `version` column: dùng cho OCC (task 2.C) — `UPDATE ... WHERE version = ?`
- 2 CHECK constraints:
  - `available_seats >= 0` → ngăn oversell ở DB level
  - `available_seats <= total_seats` → sanity check, không bao giờ có sẵn > tổng

### Table: `seats`

```sql
CREATE TABLE seats (
    id          BIGSERIAL PRIMARY KEY,
    event_id    BIGINT NOT NULL REFERENCES events(id),
    seat_label  VARCHAR(10) NOT NULL,
    status      VARCHAR(20) NOT NULL DEFAULT 'AVAILABLE',
    locked_by   UUID,
    locked_until TIMESTAMPTZ,

    CONSTRAINT chk_seat_status CHECK (status IN ('AVAILABLE', 'LOCKED', 'SOLD'))
);
```

- `status` ENUM via CHECK: AVAILABLE → LOCKED → SOLD (state machine)
- `locked_by` + `locked_until`: dùng cho Reservation strategy (2.E)
- Index trên `(event_id, status)` để query nhanh ghế còn trống

### Table: `tickets`

```sql
CREATE TABLE tickets (
    id              BIGSERIAL PRIMARY KEY,
    event_id        BIGINT NOT NULL REFERENCES events(id),
    seat_id         BIGINT NOT NULL REFERENCES seats(id),
    user_id         UUID NOT NULL,
    idempotency_key VARCHAR(255),
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT uq_ticket_event_seat UNIQUE (event_id, seat_id)
);
```

- `UNIQUE (event_id, seat_id)` → **DB-level guard chống double-book**
- `idempotency_key` → dùng cho task 3.1 (Idempotency)
- Không UNIQUE trên `idempotency_key` ở đây — check ở application layer qua Redis/filter

### Table: `reservations`

```sql
CREATE TABLE reservations (
    id              BIGSERIAL PRIMARY KEY,
    event_id        BIGINT NOT NULL REFERENCES events(id),
    seat_id         BIGINT NOT NULL REFERENCES seats(id),
    user_id         UUID NOT NULL,
    fencing_token   UUID NOT NULL,
    status          VARCHAR(20) NOT NULL DEFAULT 'PENDING',
    expires_at      TIMESTAMPTZ NOT NULL,
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),

    CONSTRAINT chk_reservation_status CHECK (status IN ('PENDING', 'CONFIRMED', 'EXPIRED', 'CANCELLED'))
);
```

- `fencing_token`: UUID sinh khi reserve — confirm phải gửi đúng token này
- `expires_at`: TTL — background job sẽ chuyển PENDING → EXPIRED nếu quá hạn
- State machine: PENDING → CONFIRMED | EXPIRED | CANCELLED

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| CHECK constraints = safety net ở DB level | CHECK không thay thế application-level concurrency control |
| UNIQUE constraint chống double-book | INSERT conflict = exception, cần handle trong code |
| `version` column sẵn sàng cho OCC | Thêm 1 column phải maintain, dù naive/pessimistic không dùng |
| Flyway = reproducible migrations | Thêm 1 tool vào stack, learning curve nhỏ |
| Normalized schema (4 tables) | JOIN nhiều hơn vs denormalized, nhưng correctness > performance |

## Latency / Throughput Analysis

Schema design ảnh hưởng đến latency qua:

1. **Index lookup**: `UNIQUE (event_id, seat_id)` trên `tickets` → B-tree index, O(log n)
2. **CHECK constraint**: evaluate in-memory, negligible overhead (<0.01ms)
3. **Foreign key check**: PG cần shared lock trên parent row khi INSERT child → potential contention nếu nhiều tickets cùng event
4. **Write amplification**: INSERT ticket = 1 row + 1 index entry + FK check + WAL write

Ước tính với 100 seats event:
- Single INSERT ticket: ~0.5ms (bao gồm FK check + UNIQUE check)
- 100 concurrent INSERTs (khác seat): ~2-5ms mỗi cái (index contention trên B-tree page)
- 100 concurrent INSERTs (cùng seat): 1 thành công, 99 fail UNIQUE violation → fast fail

## Khi nào dùng trong production

✅ Phù hợp:
- Schema với CHECK + UNIQUE → defense-in-depth, kể cả application bug cũng không double-book
- Flyway migrations → team workflow chuẩn, CI/CD friendly
- Normalized schema → khi correctness quan trọng hơn read performance

❌ Không phù hợp:
- Nếu cần horizontal sharding → FK cross-shard không work, phải denormalize
- Nếu event có 100K+ seats → single `events.available_seats` counter = bottleneck
  → Partition seats table hoặc dùng sharded counter

→ Alternative: Event Sourcing cho audit trail, CQRS cho read-heavy + write-heavy separation

## Spring Boot Implementation Notes

### Flyway config trong application.yml
```yaml
spring:
  flyway:
    enabled: true
    locations: filesystem:../../db/migrations
    baseline-on-migrate: true
  jpa:
    hibernate:
      ddl-auto: validate  # KHÔNG create — Flyway quản lý schema
```

**Gotcha:**
- `ddl-auto: validate` — Hibernate chỉ kiểm tra schema match entity, KHÔNG tạo/sửa table
- `baseline-on-migrate: true` — lần đầu chạy trên DB có sẵn, Flyway tạo baseline
- Migration file path: `filesystem:../../db/migrations` — relative từ thư mục chạy Maven
  (chạy từ `source/ticketing-service/` → `../../db/migrations/` = `db/migrations/`)

### JPA Entity mapping cần chú ý
- `@Version` annotation trên `version` column → Hibernate tự handle OCC (nhưng ta sẽ custom cho 2.C)
- `@Check` annotation (Hibernate 6.x) có thể map CHECK constraint vào entity
- `@Table(uniqueConstraints = ...)` cho UNIQUE constraint mapping
