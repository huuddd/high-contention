# 2.B — Pessimistic Locking

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích cơ chế SELECT FOR UPDATE ở PostgreSQL level — lock manager, lock modes, lock queue
- Tôi có thể giải thích tại sao blocking là trade-off chấp nhận được khi correctness là ưu tiên số 1
- Tôi có thể giải thích deadlock — circular wait, tại sao nó xảy ra với multi-row locking, cách PostgreSQL detect
- Tôi có thể tính toán latency estimate cho pessimistic locking dựa trên concurrency và transaction time
- Tôi có thể giải thích khi nào pessimistic locking là lựa chọn đúng và khi nào cần alternative

## The Mental Model

Hình dung phòng họp có **1 chìa khóa duy nhất** treo trên tường:
- Người đầu tiên đến lấy chìa khóa → vào phòng → khóa cửa
- Tất cả người sau **xếp hàng chờ** bên ngoài (FIFO queue)
- Khi người trong phòng xong → trả chìa khóa → người tiếp theo trong hàng lấy
- **Không ai vào phòng mà không có chìa khóa** → correctness 100%
- **Nhược điểm**: nếu 200 người chờ, người cuối hàng chờ rất lâu

`SELECT FOR UPDATE` chính là hành động **lấy chìa khóa**: row bị khóa cho đến khi transaction COMMIT hoặc ROLLBACK.

## Tại sao Naive bị sai — Race Condition Timeline

```
T=0ms  Thread A: SELECT available_seats FROM events WHERE id = 1
                 → available_seats = 1  ✅

T=0ms  Thread B: SELECT available_seats FROM events WHERE id = 1
                 → available_seats = 1  ✅ (cùng snapshot, cùng thấy 1!)

T=1ms  Thread A: UPDATE events SET available_seats = 0 WHERE id = 1
                 → ✅ success

T=1ms  Thread B: UPDATE events SET available_seats = -1 WHERE id = 1
                 → ❌ CHECK constraint violation HOẶC counter drift

→ Nguyên nhân gốc: SELECT và UPDATE không atomic — TOCTOU gap
→ Giải pháp: SELECT FOR UPDATE biến SELECT+UPDATE thành atomic operation
```

## Cơ chế hoạt động (DB/OS level)

### PostgreSQL Lock Manager

Khi `SELECT ... FOR UPDATE` chạy:

```
1. Parser + Planner: PostgreSQL nhận biết FOR UPDATE clause
2. Executor: tìm row theo WHERE clause
3. Lock Manager: yêu cầu RowExclusiveLock trên row
   → Nếu row chưa bị lock: cấp lock ngay, tiếp tục
   → Nếu row đã bị lock bởi txn khác: đưa txn hiện tại vào WAIT QUEUE
4. Wait Queue: FIFO — transactions chờ theo thứ tự đến
5. Khi lock holder COMMIT/ROLLBACK:
   → Lock Manager giải phóng lock
   → Transaction đầu tiên trong queue được đánh thức
   → Re-evaluate: đọc row VERSION MỚI (sau commit của holder trước)
```

### Lock Modes trong PostgreSQL

```
Lock Mode          Conflicts With         Dùng cho
─────────────────────────────────────────────────────
FOR UPDATE         FOR UPDATE,            Write intent — "tôi sẽ sửa row này"
                   FOR NO KEY UPDATE,
                   FOR SHARE,
                   FOR KEY SHARE

FOR NO KEY UPDATE  FOR UPDATE,            Sửa non-key columns
                   FOR NO KEY UPDATE,
                   FOR SHARE

FOR SHARE          FOR UPDATE,            Read lock — "không ai được sửa"
                   FOR NO KEY UPDATE

FOR KEY SHARE      FOR UPDATE             Chỉ chặn DELETE/UPDATE key columns
```

`FOR UPDATE` là mode **mạnh nhất** — chặn mọi lock mode khác trên cùng row.

### Tại sao SELECT FOR UPDATE giải quyết TOCTOU

```
T=0ms  Thread A: SELECT * FROM events WHERE id = 1 FOR UPDATE
                 → Lock acquired ✅, available_seats = 1

T=0ms  Thread B: SELECT * FROM events WHERE id = 1 FOR UPDATE
                 → ⏳ BLOCKED — chờ Thread A release lock

T=1ms  Thread A: UPDATE events SET available_seats = 0 WHERE id = 1
T=2ms  Thread A: INSERT ticket, COMMIT → Lock released

T=2ms  Thread B: Lock acquired ✅ — RE-READS row → available_seats = 0
                 → available_seats <= 0 → return SEAT_NOT_AVAILABLE

→ Không có TOCTOU gap: SELECT và UPDATE nằm trong cùng lock scope
→ Thread B thấy giá trị SAU khi Thread A commit — không phải snapshot cũ
```

### WAL (Write-Ahead Log) và Lock Persistence

```
Lock information KHÔNG ghi vào WAL — lock chỉ tồn tại trong shared memory.
Nếu PostgreSQL crash giữa transaction:
  → Lock tự động giải phóng (in-memory state lost)
  → Uncommitted changes bị rollback (WAL recovery)
  → Không có "stale lock" sau crash — safe by design
```

## SQL / Implementation

```sql
-- Bước 1: Lock event row — tất cả concurrent transactions xếp hàng ở đây
SELECT * FROM events WHERE id = :eventId FOR UPDATE;

-- Bước 2: Check available seats (giá trị đã up-to-date vì đang hold lock)
-- Nếu available_seats <= 0 → return ngay, COMMIT (release lock)

-- Bước 3: Lock seat row — tránh double-book
SELECT * FROM seats
WHERE event_id = :eventId AND seat_label = :seatLabel
FOR UPDATE;

-- Bước 4: Decrement counter
UPDATE events SET available_seats = available_seats - 1 WHERE id = :eventId;

-- Bước 5: Mark seat as SOLD
UPDATE seats SET status = 'SOLD' WHERE id = :seatId;

-- Bước 6: Create ticket
INSERT INTO tickets (event_id, seat_id, user_id) VALUES (:eventId, :seatId, :userId);

-- Bước 7: COMMIT → release ALL locks
COMMIT;
```

**Quan trọng: Lock CÙNG LÚC event + seat**
- Lock event trước (coarse-grained) → serialize access vào counter
- Lock seat sau (fine-grained) → prevent double-book
- Thứ tự lock phải **consistent** across tất cả code paths → tránh deadlock

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Correctness 100% — oversell = 0 | Blocking — throughput giảm tuyến tính với contention |
| Đơn giản, dễ hiểu và audit | Deadlock risk khi lock nhiều rows theo thứ tự khác nhau |
| Không cần retry logic phức tạp | Connection pool exhaustion dưới spike load |
| PostgreSQL native — không cần external tool | Lock contention tăng phi tuyến tính khi concurrency tăng |
| Crash-safe — lock auto-release | Không scale horizontally — lock chỉ work trong 1 DB instance |

## Latency / Throughput Analysis

### Công thức ước lượng

```
Giả sử:
  C = concurrent threads
  T = transaction time (bao gồm cả business logic)
  
Vì tất cả threads serialize qua 1 lock:
  Throughput = 1 / T
  Average wait time = (C - 1) × T / 2
  p50 latency ≈ C/2 × T
  p95 latency ≈ C × 0.95 × T
  p99 latency ≈ C × 0.99 × T
```

### Ví dụ cụ thể

```
C = 200 concurrent threads, T = 5ms (fast DB operation):
  Throughput = 200 TPS (serialize 200 requests qua 5ms mỗi cái)
  Average wait = 199 × 5ms / 2 ≈ 497ms
  p50 ≈ 100 × 5ms = 500ms
  p95 ≈ 190 × 5ms = 950ms
  p99 ≈ 198 × 5ms = 990ms

C = 200, T = 200ms (có gọi payment gateway):
  Throughput = 5 TPS 😱
  p50 ≈ 100 × 200ms = 20 SECONDS
  p95 ≈ 190 × 200ms = 38 SECONDS
  → User sẽ timeout, abandon purchase
```

→ Pessimistic locking **gãy** khi `C × T` > vài giây.
→ Nếu transaction time > 50ms, cần chuyển sang Reservation (2.E) hoặc Queue (2.F).

## Real-World Context

### Stripe Payments — tại sao họ KHÔNG dùng pessimistic locking cho main flow
- Payment processing: T = 200-2000ms (gọi card network)
- Dùng pessimistic lock = serialize tất cả payments qua 1 lock → throughput ~1 TPS
- Thay vào đó: idempotency keys + OCC + eventual consistency

### Airline Booking Systems (GDS: Amadeus, Sabre)
- Dùng pessimistic locking cho **seat map update** (T < 5ms)
- KHÔNG dùng cho full booking flow (T > 30s) — dùng reservation + timeout

### PostgreSQL Advisory Locks — alternative nhẹ hơn
- `pg_advisory_lock(event_id)`: lock level application, không lock row
- Nhẹ hơn FOR UPDATE, nhưng khó quản lý và debug

## Ordered Locking — tránh deadlock khi cần lock nhiều rows

### Deadlock scenario

```
T=0ms  Thread A: SELECT * FROM events WHERE id=1 FOR UPDATE → ✅ lock event 1
T=0ms  Thread B: SELECT * FROM events WHERE id=2 FOR UPDATE → ✅ lock event 2

T=1ms  Thread A: SELECT * FROM events WHERE id=2 FOR UPDATE → ⏳ chờ B release event 2
T=1ms  Thread B: SELECT * FROM events WHERE id=1 FOR UPDATE → ⏳ chờ A release event 1

→ DEADLOCK: A chờ B, B chờ A — circular wait
→ PostgreSQL detect sau deadlock_timeout (1s) → abort 1 transaction
```

### Giải pháp: Ordered Locking

```
Quy tắc: LUÔN lock theo thứ tự ID tăng dần.

Thread A cần lock event 1 + event 2: lock(1) → lock(2)
Thread B cần lock event 2 + event 1: lock(1) → lock(2)  ← CÙNG THỨ TỰ

→ Không có circular wait → không deadlock
```

Trong project này: chỉ lock 1 event + 1 seat → deadlock risk thấp.
Nhưng nếu mở rộng (multi-event booking): **ordered locking là bắt buộc**.

## Khi nào dùng trong production

✅ Phù hợp:
- **Single-DB, low-to-medium contention** (< 500 concurrent)
- **Short transactions** (T < 10ms) — không gọi external service trong lock
- **Correctness > throughput** — financial operations, inventory management
- **Simple audit requirement** — dễ explain "row bị lock cho đến khi xong"

❌ Không phù hợp:
- **High concurrency** (> 1000 concurrent) — p95 latency quá cao
- **Long transactions** (gọi payment gateway, external API) — connection pool exhaustion
- **Multi-DB / distributed** — row lock không cross DB instances
- **Spike traffic** (flash sale) — lock queue grow unbounded

→ Nếu T > 50ms: chuyển sang **Reservation + Fencing (2.E)** — tách payment khỏi lock window
→ Nếu C > 1000: chuyển sang **Queue-based (2.F)** — serialize ở application level

## Spring Boot Implementation Notes

### @Transactional configuration
```java
@Transactional  // default: READ_COMMITTED, REQUIRED propagation
```
- **KHÔNG cần** thay đổi isolation level — FOR UPDATE tự handle locking
- `@Transactional` chỉ đảm bảo transaction boundary — lock released khi method return

### Gotchas quan trọng
1. **Lock scope = transaction scope** — nếu `@Transactional` wraps cả method,
   lock giữ từ SELECT FOR UPDATE đến COMMIT. Giữ transaction ngắn!
2. **Lazy loading trong lock scope** — nếu JPA lazy-load entity sau lock,
   thêm query = thêm thời gian giữ lock. Dùng eager fetch hoặc native query.
3. **Exception = rollback = lock release** — nếu throw exception trong `@Transactional`,
   Spring rollback → lock released ngay. Đây là behavior đúng.
4. **Lock timeout** — PostgreSQL default: chờ vô hạn. Nên set `statement_timeout`
   hoặc `lock_timeout` trong connection pool để tránh thread chờ mãi:
   ```sql
   SET lock_timeout = '5s';
   ```
