# 2.C — Optimistic Concurrency Control (OCC) + Retry Backoff

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích OCC — "giả sử không ai xung đột, kiểm tra lại lúc commit"
- Tôi có thể giải thích tại sao version column là cơ chế detection đơn giản nhất cho OCC
- Tôi có thể giải thích exponential backoff + jitter — tại sao cần cả hai, không chỉ backoff
- Tôi có thể so sánh OCC vs Pessimistic: khi nào OCC tốt hơn, khi nào tệ hơn
- Tôi có thể tính throughput/latency cho OCC dưới các mức contention khác nhau

## The Mental Model

Hình dung **nhiều người cùng edit 1 Google Doc** (không có real-time sync):
1. Mỗi người **tải về bản sao** (READ + ghi nhớ version number)
2. Mỗi người **edit bản sao của mình** (không biết người khác đang edit gì)
3. Khi **lưu lại**, hệ thống kiểm tra: "version trên server vẫn giống lúc tải?"
   - **Giống** → lưu thành công, tăng version lên 1
   - **Khác** → "Ai đó đã sửa trước bạn!" → tải lại bản mới, sửa lại, thử lưu lại

Khác biệt với Pessimistic:
- **Pessimistic** = khóa file: "Không ai được mở cho đến khi tôi xong"
- **OCC** = edit tự do: "Cứ edit đi, kiểm tra xung đột khi lưu"

→ OCC **optimistic** vì giả sử xung đột hiếm. Nếu xung đột thực sự hiếm → throughput cao hơn Pessimistic.
→ Nếu xung đột nhiều → retry storm → throughput **tệ hơn** Pessimistic.

## Tại sao Pessimistic chưa đủ

```
Pessimistic Locking — vấn đề:

C = 200 concurrent threads, T = 5ms
→ Tất cả 200 threads xếp hàng qua 1 lock
→ p95 ≈ 190 × 5ms = 950ms
→ Throughput = 200 TPS (serialized)

Vấn đề: BLOCKING — 199 threads CHẲNG LÀM GÌ, chỉ chờ lock.
Connection pool (max 20) → chỉ 20 threads chờ lock → 180 threads chờ connection → timeout

OCC approach:
→ Không blocking — tất cả 200 threads chạy song song
→ Khi conflict (version mismatch) → retry với backoff
→ Dưới low contention: throughput >> Pessimistic
→ Dưới high contention: retry storm, nhưng ít nhất connection pool không bị exhaustion
```

## Cơ chế hoạt động (DB/OS level)

### Version-based OCC — cách PostgreSQL xử lý

```
Mỗi event row có: version BIGINT (bắt đầu từ 0)

Thread A:                              Thread B:
─────────                              ─────────
SELECT * FROM events WHERE id=1        SELECT * FROM events WHERE id=1
→ available=10, version=5              → available=10, version=5

(Cả hai thấy cùng version — không lock)

UPDATE events                          UPDATE events
SET available_seats = 9,               SET available_seats = 9,
    version = 6                            version = 6
WHERE id=1 AND version=5              WHERE id=1 AND version=5

Thread A chạy trước:                   Thread B chạy sau:
→ WHERE id=1 AND version=5 ✅         → WHERE id=1 AND version=5 ❌
→ 1 row affected → SUCCESS            → version đã = 6 (A đã commit)
                                       → 0 rows affected → CONFLICT
                                       → Retry: đọc lại, version=6
                                       → UPDATE WHERE version=6 → ✅
```

### Tại sao "0 rows affected" = conflict detection

```sql
UPDATE events
SET available_seats = available_seats - 1, version = version + 1
WHERE id = :id AND version = :version AND available_seats > 0
```

Kết quả:
- `affected_rows = 1`: update thành công, không ai sửa giữa chừng
- `affected_rows = 0`: có 2 khả năng:
  a) version mismatch — ai đó đã update → RETRY
  b) available_seats = 0 — hết vé → STOP (no retry)

→ Phân biệt 2 case: SELECT lại để check available_seats.
  Nếu available_seats > 0 nhưng version khác → conflict → retry.
  Nếu available_seats = 0 → sold out → return SEAT_NOT_AVAILABLE.

### PostgreSQL MVCC + OCC interaction

```
OCC KHÔNG thay đổi PostgreSQL behavior — vẫn dùng READ COMMITTED.
OCC chỉ thêm 1 WHERE condition: AND version = ?

Khi UPDATE chạy dưới READ COMMITTED:
1. PG tìm row matching WHERE clause
2. Nếu row đang bị locked bởi txn khác → WAIT (ngắn — chỉ đợi commit)
3. Sau khi txn khác commit → PG re-evaluate WHERE clause trên row mới
4. Nếu version đã thay đổi → WHERE clause fails → 0 rows affected
5. Application detect 0 rows → retry

Quan trọng: OCC vẫn có 1 khoảng wait ngắn ở bước 2 (wait for concurrent commit).
Nhưng wait này NGẮN HƠN NHIỀU so với Pessimistic (chỉ wait cho commit, không wait cho toàn bộ transaction).
```

## SQL / Implementation

```sql
-- Bước 1: READ — không lock, không FOR UPDATE
SELECT id, available_seats, version FROM events WHERE id = :eventId;
-- → available_seats = 10, version = 5

-- Bước 2: Conditional UPDATE — atomic check-and-set
UPDATE events
SET available_seats = available_seats - 1, version = version + 1
WHERE id = :eventId AND version = 5 AND available_seats > 0;
-- → affected_rows = 1? SUCCESS : CONFLICT (version changed)

-- Bước 3: Nếu CONFLICT → retry với backoff
-- Retry 1: wait 50ms + jitter → SELECT lại → UPDATE WHERE version = 6
-- Retry 2: wait 100ms + jitter → SELECT lại → UPDATE WHERE version = 7
-- ...
-- Retry 5: fail → return MAX_RETRIES_EXCEEDED
```

### Seat-level OCC

```sql
-- Seat không có version column — dùng status check thay thế
UPDATE seats SET status = 'SOLD'
WHERE id = :seatId AND status = 'AVAILABLE';
-- → affected_rows = 1? SUCCESS : SEAT_ALREADY_SOLD
```

Status-based OCC: `WHERE status = 'AVAILABLE'` đóng vai trò tương tự version check.
Nếu seat đã bị sold → 0 rows → detect conflict.

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Non-blocking reads — không giữ lock khi đọc | Retry storm khi contention cao |
| Throughput cao khi contention thấp | Throughput giảm mạnh khi contention tăng |
| Không deadlock (không lock rows) | Wasted work: transaction chạy xong rồi mới biết conflict |
| Connection pool không bị exhaustion | Latency không predictable (phụ thuộc retry count) |
| Scale tốt cho read-heavy workload | Không fair — thread may mắn thắng, thread xui thua |

## Latency / Throughput Analysis

### Low contention (10 concurrent, 100 seats)

```
Conflict probability per request ≈ 10/100 = 10%
Average retries per request ≈ 0.1
Effective throughput ≈ 10 / (5ms × 1.1) ≈ 1800 TPS
p50 ≈ 5ms (no retry needed)
p95 ≈ 55ms (1 retry + backoff)

→ OCC >> Pessimistic (Pessimistic p95 ≈ 25ms, nhưng throughput chỉ 200 TPS)
```

### High contention (200 concurrent, 1 seat)

```
Conflict probability per request ≈ 199/200 = 99.5%
Average retries per request ≈ 4.5 (gần max)
Nhiều requests fail sau max retries
Effective throughput ≈ 1 TPS (chỉ 1 thread thắng)
p50 ≈ timeout (hầu hết fail)

→ OCC << Pessimistic (Pessimistic vẫn đảm bảo fairness — FIFO queue)
```

### Crossover point

```
OCC tốt hơn Pessimistic khi: conflict_rate < ~30%
Pessimistic tốt hơn OCC khi: conflict_rate > ~30%

Quy tắc ngón tay cái:
  Nếu C / available_resources < 0.3 → dùng OCC
  Nếu C / available_resources > 0.3 → dùng Pessimistic hoặc Queue
```

## Real-World Context

### Git — OCC ở application level
- `git push`: kiểm tra remote HEAD có thay đổi không
- Nếu thay đổi → reject → user phải pull + merge + push lại
- OCC hoạt động tuyệt vời vì conflict rate thấp (2 người ít khi sửa cùng file)

### CAS (Compare-And-Swap) — CPU level OCC
- `CMPXCHG` instruction: if (memory == expected) then memory = new_value
- Lock-free data structures dùng CAS loop — chính là OCC ở hardware level
- Java `AtomicInteger.compareAndSet()` = CAS = OCC

### Amazon DynamoDB — OCC nổi tiếng nhất
- `ConditionExpression`: `attribute_not_exists(pk) OR version = :expected_version`
- DynamoDB KHÔNG hỗ trợ locking → OCC là lựa chọn duy nhất
- Hoạt động tốt vì DynamoDB workload thường low contention per item

## Khi nào dùng trong production

✅ Phù hợp:
- **Read-heavy, write-light** — ít conflict → ít retry
- **Low-to-medium contention** — conflict rate < 30%
- **Distributed systems** — không cần distributed lock
- **Latency-sensitive reads** — không block đọc

❌ Không phù hợp:
- **Hot key / hot row** — 1 row bị 1000 threads cùng update → retry storm
- **Flash sale / Stock=1** — gần 100% conflict → OCC gần như unusable
- **Fairness required** — OCC không FIFO, thread may mắn thắng
- **Long transactions** — wasted work khi conflict detected muộn

→ Nếu contention cao: chuyển sang **Pessimistic (2.B)** hoặc **Queue (2.F)**
→ Nếu cần fairness: chuyển sang **Queue (2.F)**

## Spring Boot Implementation Notes

### OCC trong Spring — KHÔNG dùng @Version

```java
// KHÔNG dùng JPA @Version vì:
// 1. @Version tự động throw OptimisticLockException — mất control
// 2. Không custom được retry logic
// 3. Khó integrate với ConflictMetrics

// THAY VÀO ĐÓ: manual version check trong native query
@Modifying
@Query(value = "UPDATE events SET available_seats = available_seats - 1, " +
    "version = version + 1 WHERE id = :id AND version = :version AND available_seats > 0",
    nativeQuery = true)
int decrementAvailableSeatsWithVersion(@Param("id") Long id, @Param("version") Long version);
```

### Retry integration với RetryWithBackoff

```java
return retryWithBackoff.execute(
    () -> attemptPurchase(eventId, userId, seatLabel),
    OccConflictException.class,  // chỉ retry khi version mismatch
    "occ"                        // strategy name cho metrics
);
```

### Transaction boundary — mỗi retry = transaction MỚI

```java
// QUAN TRỌNG: @Transactional KHÔNG wrap retry loop
// Mỗi attempt = 1 transaction riêng
// Nếu @Transactional wrap cả retry → tất cả retries trong 1 txn → sai

// ĐÚng:
public TicketResult reserveAndBuy(...) {  // KHÔNG @Transactional ở đây
    return retryWithBackoff.execute(
        () -> doAttempt(eventId, userId, seatLabel),  // @Transactional ở doAttempt
        ...
    );
}
```
