# 1.1 — Docker Compose + Makefile

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích tại sao cần container hóa PostgreSQL và Redis cho local dev
- Tôi có thể giải thích health check hoạt động như thế nào trong Docker Compose
- Tôi có thể giải thích tại sao Makefile là abstraction layer cần thiết giữa developer và toolchain
- Tôi có thể giải thích Docker networking: service name = hostname trong cùng network

## The Mental Model

Hình dung Docker Compose như một "phòng lab di động":
- Mỗi service (PostgreSQL, Redis) là một máy riêng biệt trong cùng phòng lab
- Chúng nói chuyện với nhau qua tên (hostname = service name), không cần biết IP
- Health check giống như "kiểm tra máy đã sẵn sàng chưa" trước khi bắt đầu thí nghiệm
- Makefile là "bảng điều khiển" — developer chỉ cần nhấn nút, không cần nhớ câu lệnh

## Tại sao cần Docker Compose

Không dùng Docker Compose:
- Developer A dùng PostgreSQL 14, Developer B dùng 16 → behavior khác nhau
- SERIALIZABLE isolation có sự khác biệt giữa PG versions
- Redis version khác nhau → command syntax có thể thay đổi
- "Chạy trên máy tôi được" → classic environment mismatch

Dùng Docker Compose:
- **Reproducibility**: mọi người chạy cùng PG 16.x + Redis 7.x
- **Isolation**: không conflict với database local đang có
- **One-command setup**: `make db-up` thay vì 10 bước manual
- **CI/CD ready**: cùng compose file chạy trên CI

## Cơ chế hoạt động

### Docker Compose Networking
- Compose tạo 1 bridge network mặc định: `ticketing-net`
- Mỗi service join vào network này với hostname = service name
- `postgres:5432` từ trong container, `localhost:5432` từ host
- Spring Boot app chạy trên host → kết nối qua `localhost:5432`

### Health Check
```yaml
healthcheck:
  test: ["CMD-SHELL", "pg_isready -U ticketing"]
  interval: 5s
  timeout: 3s
  retries: 5
```
- Docker daemon gọi `pg_isready` mỗi 5s
- Nếu exit code = 0 → healthy
- Nếu fail 5 lần liên tiếp → unhealthy
- Service khác có `depends_on: condition: service_healthy` sẽ chờ

### Makefile — Why Not Just Shell Scripts?
- **Discoverability**: `make <tab>` hiện tất cả targets
- **Idempotency**: Make tự track dependency (dù ở đây ta dùng .PHONY)
- **Convention**: hầu hết open-source projects dùng Makefile
- **Cross-platform**: chạy được trên Linux, macOS, Windows (WSL/Git Bash)

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Reproducible environment | Docker overhead (~100MB RAM cho PG+Redis) |
| One-command setup | Developer cần cài Docker Desktop |
| Version pinning (PG 16, Redis 7) | Volume mount có thể slow trên macOS |
| Network isolation | Port conflict nếu PG local đã chạy |

## Cấu hình chi tiết

### PostgreSQL 16
- **Port**: 5432 (host) → 5432 (container)
- **Database**: `ticketing`
- **User/Password**: `ticketing` / `ticketing` (dev only, KHÔNG dùng production)
- **Volume**: `pgdata` — persist data giữa restart
- **Tuning**: `max_connections=200` để support load test concurrent connections

### Redis 7
- **Port**: 6379 (host) → 6379 (container)
- **No auth** (dev only)
- **Dùng cho**: Queue-based strategy (2.F), Idempotency cache (3.1)

### Makefile Targets
| Target | Mô tả |
|--------|--------|
| `db-up` | Start PostgreSQL + Redis |
| `db-down` | Stop and remove containers |
| `db-reset` | Stop, remove volumes, restart |
| `migrate` | Run Flyway migrations |
| `seed` | Insert test data |
| `app-run` | Start Spring Boot app |
| `bench:naive` | k6 run naive benchmark |
| `bench:pessimistic` | k6 run pessimistic benchmark |
| `bench:occ` | k6 run OCC benchmark |
| `bench:serializable` | k6 run serializable benchmark |
| `bench:reservation` | k6 run reservation benchmark |
| `bench:queue` | k6 run queue benchmark |

## Khi nào dùng trong production

✅ Phù hợp:
- Local development — đảm bảo team dùng cùng version
- CI/CD pipeline — spin up services cho integration test
- Demo environment — quick setup cho stakeholders

❌ Không phù hợp:
- Production deployment — dùng managed services (RDS, ElastiCache)
- High-availability setup — Compose không có orchestration (dùng K8s)

→ Alternative cho production: Terraform + AWS RDS + ElastiCache hoặc K8s Helm charts

## Spring Boot Connection Notes

```yaml
spring:
  datasource:
    url: jdbc:postgresql://localhost:5432/ticketing
    username: ticketing
    password: ticketing
    hikari:
      maximum-pool-size: 50    # phải < max_connections PG (200)
      minimum-idle: 10
      connection-timeout: 5000  # 5s — fail fast under load
  data:
    redis:
      host: localhost
      port: 6379
```

**Gotcha cần biết:**
- HikariCP pool size phải nhỏ hơn PG `max_connections` — nếu bằng hoặc lớn hơn, connection refused
- `connection-timeout` nên ngắn (5s) để fail fast, không để thread chờ mãi
- Khi chạy load test 200 VUs, mỗi VU giữ 1 connection → cần pool size >= 50
  (k6 VU không map 1:1 với DB connection vì Spring xử lý request tuần tự trên thread pool)
