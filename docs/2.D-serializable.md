# 2.D — SERIALIZABLE Isolation Level

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích SERIALIZABLE isolation trong PostgreSQL — SSI (Serializable Snapshot Isolation)
- Tôi có thể giải thích sự khác biệt giữa true serializable (2PL) và SSI (optimistic approach)
- Tôi có thể giải thích tại sao SERIALIZABLE trong PostgreSQL KHÔNG blocking như Pessimistic — nó detect conflict và abort
- Tôi có thể so sánh SERIALIZABLE vs OCC: cùng optimistic mindset, khác ở level detection
- Tôi có thể giải thích serialization anomalies: write skew, phantom reads

## The Mental Model

Hình dung **3 kế toán viên** cùng làm sổ sách cho 1 công ty:
- **READ COMMITTED** (Naive): mỗi kế toán nhìn bảng cân đối MỚI NHẤT trước mỗi dòng ghi — nhưng có thể thấy inconsistency giữa các dòng
- **OCC**: mỗi kế toán ghi nhớ "phiên bản" lúc đọc, kiểm tra lại khi lưu — nhưng chỉ kiểm tra 1 row cụ thể
- **SERIALIZABLE**: PostgreSQL đảm bảo KẾT QUẢ giống như 3 kế toán **làm tuần tự** (A xong → B → C), dù thực tế họ chạy song song

SERIALIZABLE = "nếu kết quả concurrent execution khác kết quả serial execution → abort 1 transaction"

**Khác biệt quan trọng với OCC (2.C):**
- OCC: application tự quản lý version column, tự detect conflict
- SERIALIZABLE: **PostgreSQL tự detect** mọi serialization anomaly — application chỉ cần retry khi bị abort
- SERIALIZABLE bắt được nhiều anomaly hơn OCC: write skew, phantom reads — những thứ version check đơn giản bỏ sót

## Tại sao OCC chưa đủ

```
OCC (2.C) — limitation:

Thread A: SELECT available_seats=10, version=5
          UPDATE events SET available_seats=9, version=6 WHERE version=5 ✅

Thread B: SELECT count(*) FROM tickets WHERE event_id=1
          → thấy count=90 (đúng tại snapshot)
          INSERT ticket (based on count check)

OCC chỉ check version trên event row — KHÔNG detect:
- Thread B đọc tickets table → kết quả có thể stale
- Write skew: 2 threads đọc overlapping data, ghi non-overlapping rows
  → mỗi thread thấy state hợp lệ, nhưng combined effect vi phạm invariant

SERIALIZABLE bắt tất cả: vì PG tracks mọi read dependency, không chỉ 1 version column.
```

## Cơ chế hoạt động (DB/OS level)

### PostgreSQL SSI — Serializable Snapshot Isolation

```
PostgreSQL KHÔNG dùng traditional 2PL (Two-Phase Locking) cho SERIALIZABLE.
Thay vào đó: SSI — extension của MVCC snapshot isolation.

Cơ chế 3 bước:
1. Mỗi transaction chạy trên snapshot riêng (giống READ COMMITTED)
2. PostgreSQL tracks "rw-dependencies" (read-write dependencies):
   - T1 đọc row X → T2 ghi row X → dependency: T1 →rw T2
   - T2 đọc row Y → T1 ghi row Y → dependency: T2 →rw T1
3. Nếu phát hiện CYCLE trong dependency graph → "dangerous structure"
   → Abort 1 transaction với: ERROR 40001 (serialization_failure)
   → Application phải RETRY toàn bộ transaction
```

### SSI vs 2PL — tại sao PostgreSQL chọn SSI

```
2PL (Traditional SERIALIZABLE — MySQL InnoDB):
  + Ngăn anomaly bằng LOCK — không cần retry
  - Blocking: read locks + write locks → deadlock risk
  - Throughput thấp: readers block writers, writers block readers
  
SSI (PostgreSQL):
  + Non-blocking reads: readers KHÔNG block writers (snapshot isolation)
  + Throughput cao hơn 2PL cho read-heavy workload
  - Cần retry: aborted transactions phải chạy lại
  - False positives: đôi khi abort transaction không thực sự conflict
    (vì SSI detect "dangerous structure", không phải actual anomaly)
```

### Serialization Failure — khi nào xảy ra

```
Scenario: 2 threads cùng mua vé cho event 1

T1: BEGIN ISOLATION LEVEL SERIALIZABLE
    SELECT * FROM events WHERE id = 1  → available = 10 (snapshot A)
    
T2: BEGIN ISOLATION LEVEL SERIALIZABLE  
    SELECT * FROM events WHERE id = 1  → available = 10 (snapshot B)
    
T1: UPDATE events SET available_seats = 9 WHERE id = 1  → ✅
T1: INSERT INTO tickets (...) → ✅
T1: COMMIT → ✅

T2: UPDATE events SET available_seats = 9 WHERE id = 1
    → PostgreSQL detects: T2 read data that T1 modified
    → rw-dependency cycle: T2 →rw T1 →rw T2
    → ERROR: could not serialize access due to concurrent update
    → T2 must RETRY

Sau retry:
T2': BEGIN ISOLATION LEVEL SERIALIZABLE
     SELECT * FROM events WHERE id = 1 → available = 9 (new snapshot)
     UPDATE events SET available_seats = 8 WHERE id = 1 → ✅
     INSERT INTO tickets (...) → ✅
     T2': COMMIT → ✅
```

### PSQLException error codes

```
40001 — serialization_failure
  → "could not serialize access due to concurrent update"
  → Retry-safe: transaction had no side effects (was rolled back)
  
40P01 — deadlock_detected  
  → Rất hiếm với SSI (SSI không dùng locks), nhưng possible
  → Cũng retry-safe
```

## SQL / Implementation

```sql
-- Bước 1: Set isolation level
SET TRANSACTION ISOLATION LEVEL SERIALIZABLE;
-- (Trong Spring: @Transactional(isolation = Isolation.SERIALIZABLE))

-- Bước 2: Plain SELECT — PostgreSQL tracks read dependencies
SELECT * FROM events WHERE id = :eventId;
-- SSI tracks: "T này đã đọc events row id=1"

-- Bước 3: Plain SELECT seat
SELECT * FROM seats WHERE event_id = :eventId AND seat_label = :seatLabel;
-- SSI tracks: "T này đã đọc seats row"

-- Bước 4: UPDATE — nếu có conflict, PG sẽ abort TẠI ĐÂY
UPDATE events SET available_seats = available_seats - 1 WHERE id = :eventId;

-- Bước 5: UPDATE seat
UPDATE seats SET status = 'SOLD' WHERE id = :seatId;

-- Bước 6: INSERT ticket
INSERT INTO tickets (event_id, seat_id, user_id) VALUES (:eventId, :seatId, :userId);

-- Bước 7: COMMIT — hoặc PG đã abort trước đó
COMMIT;
```

**Quan trọng:** Code SQL giống hệt Naive (2.A)! Chỉ khác ở `ISOLATION LEVEL SERIALIZABLE`.
→ PostgreSQL tự detect anomaly — application không cần version column hay FOR UPDATE.

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Strongest correctness guarantee — catches ALL anomalies | Serialization failures cần retry |
| Non-blocking reads (SSI, not 2PL) | False positives — abort khi không thực sự conflict |
| Code đơn giản nhất — no version column, no FOR UPDATE | Retry overhead: wasted work khi abort |
| PostgreSQL tự detect — ít bug hơn manual OCC | Throughput giảm khi contention tăng (giống OCC) |
| Catches write skew — OCC manual không detect được | Memory overhead: PG tracks rw-dependencies |
| Portable concept — mọi DB có SERIALIZABLE (dù impl khác) | PostgreSQL-specific behavior (SSI vs 2PL) |

## Latency / Throughput Analysis

### So sánh với OCC (2.C)

```
SERIALIZABLE vs OCC dưới cùng load (200 concurrent, 100 seats):

OCC:
  - Conflict detection: application-level (version check)
  - Conflict scope: chỉ event.version
  - False positive rate: 0% (chính xác)
  - Retry rate: ~conflict rate

SERIALIZABLE:
  - Conflict detection: PostgreSQL SSI engine
  - Conflict scope: toàn bộ read/write set
  - False positive rate: ~5-15% (PG over-cautious)
  - Retry rate: conflict rate + false positives

→ SERIALIZABLE có retry rate CAO HƠN OCC ~10-15%
→ Nhưng SERIALIZABLE catches write skew mà OCC bỏ sót
```

### Khi nào SERIALIZABLE thua OCC

```
Simple counter update (chỉ 1 row):
  OCC: version check đủ rồi → 0% false positive
  SERIALIZABLE: SSI tracks nhiều hơn cần thiết → ~5% false positive → more retries

→ Cho use case ticketing đơn giản, OCC đủ tốt và ít retry hơn
```

### Khi nào SERIALIZABLE thắng OCC

```
Complex invariant (multiple tables):
  "Tổng tickets + active reservations <= total_seats"
  
  OCC: phải manual version-check trên events, tickets, reservations → phức tạp, dễ bug
  SERIALIZABLE: PostgreSQL tự detect — code vẫn đơn giản
  
→ Cho invariant phức tạp, SERIALIZABLE ít bug hơn
```

## Real-World Context

### CockroachDB — SERIALIZABLE by default
- CockroachDB dùng SERIALIZABLE as DEFAULT isolation level
- Lý do: distributed system → consistency quan trọng hơn throughput
- Application PHẢI handle serialization retries — đây là contract

### PostgreSQL — READ COMMITTED by default
- Hầu hết PG applications dùng READ COMMITTED — đủ cho most use cases
- SERIALIZABLE reserved cho: financial ledgers, audit trails, complex invariants
- Production tip: nên test với SERIALIZABLE trong CI, deploy với READ COMMITTED + manual locking

### Google Spanner — external consistency
- Spanner cung cấp "external consistency" — stronger than SERIALIZABLE
- Dùng TrueTime API (GPS + atomic clocks) để order transactions globally
- Cost: latency ~10ms per write (TrueTime confidence interval)

## Khi nào dùng trong production

✅ Phù hợp:
- **Complex multi-table invariants** — SERIALIZABLE catches everything
- **Rapid prototyping** — code đơn giản nhất, để PG lo concurrency
- **Audit-critical** — cần strongest guarantee
- **Read-heavy with occasional writes** — SSI non-blocking reads

❌ Không phù hợp:
- **Simple single-row contention** — OCC + version check đủ, ít retry hơn
- **High write contention** — serialization failures storm
- **Cross-DB consistency** — SERIALIZABLE chỉ work trong 1 PG instance
- **Latency-sensitive** — retry adds unpredictable latency

→ Nếu invariant đơn giản (1 counter): dùng **OCC (2.C)** — ít retry hơn
→ Nếu cần deterministic latency: dùng **Pessimistic (2.B)** hoặc **Queue (2.F)**

## Spring Boot Implementation Notes

### @Transactional với SERIALIZABLE

```java
@Transactional(isolation = Isolation.SERIALIZABLE)
public TicketResult doAttempt(...) {
    // Code GIỐNG HỆT Naive — chỉ thêm isolation level
    // PostgreSQL tự detect serialization anomaly
}
```

### Retry — giống OCC nhưng catch exception khác

```java
// OCC: catch OccConflictException (custom, application-level)
// SERIALIZABLE: catch PSQLException with SQLState 40001 (database-level)

// Spring wraps PSQLException → CannotSerializeTransactionException
// hoặc PessimisticLockingFailureException (confusing name!)
```

### Gotcha: Spring exception mapping

```java
// PostgreSQL 40001 → Spring maps to:
//   org.springframework.dao.CannotSerializeTransactionException
//   (subclass of PessimisticLockingFailureException — tên CONFUSING)
//
// Cần catch cả hai:
//   CannotSerializeTransactionException — serialization failure
//   PessimisticLockingFailureException — parent class, catches deadlocks too
```

### Transaction boundary — giống OCC

```java
// QUAN TRỌNG: retry loop NGOÀI @Transactional
// Mỗi retry = transaction MỚI = snapshot MỚI
public TicketResult reserveAndBuy(...) {  // KHÔNG @Transactional
    return retryWithBackoff.execute(
        () -> doAttempt(...),  // @Transactional(isolation = SERIALIZABLE)
        ...
    );
}
```
