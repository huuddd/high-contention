# 1.4 — Shared Utilities

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích Strategy Pattern và tại sao nó cho phép swap concurrency strategy mà không sửa controller
- Tôi có thể giải thích exponential backoff + jitter — tại sao cần jitter để tránh thundering herd
- Tôi có thể giải thích tại sao Micrometer counters quan trọng cho observability trong high-contention system
- Tôi có thể giải thích idempotency — tại sao `Idempotency-Key` header cần thiết khi retry có thể xảy ra
- Tôi có thể giải thích tại sao shared utilities phải ở `main` branch — dùng chung cho tất cả strategies

## The Mental Model

Shared utilities giống như **bộ công cụ tiêu chuẩn** trong phòng lab:
- **TicketingStrategy** = ổ cắm tiêu chuẩn — mọi thiết bị (strategy A-F) cắm vào cùng 1 ổ
- **RetryWithBackoff** = timer thông minh — khi thí nghiệm thất bại, chờ lâu hơn trước khi thử lại, cộng thêm random để tránh tất cả thử lại cùng lúc
- **ConflictMetrics** = đồng hồ đo — đếm bao nhiêu lần xung đột, bao nhiêu lần retry, bao nhiêu deadlock
- **IdempotencyFilter** = bộ lọc trùng — nếu cùng 1 yêu cầu gửi 2 lần, chỉ xử lý 1 lần

## Tại sao cần shared utilities trước khi implement strategies

```
Không có shared utilities:
  Strategy A: copy paste retry logic
  Strategy B: copy paste retry logic (khác chút)
  Strategy C: viết lại retry logic (bug khác)
  → 3 bản retry khác nhau, 3 bộ metrics khác nhau, benchmark không so sánh được

Có shared utilities:
  Strategy A-F: dùng cùng RetryWithBackoff, cùng ConflictMetrics
  → Benchmark so sánh táo-với-táo: cùng retry config, cùng metrics
  → Bug fix 1 chỗ = fix cho tất cả strategies
```

## Cơ chế hoạt động

### 1. Strategy Pattern — TicketingStrategy interface

```java
public interface TicketingStrategy {
    TicketResult reserveAndBuy(Long eventId, UUID userId, String seatLabel);
}
```

**Tại sao Strategy Pattern?**
- Controller chỉ gọi `strategy.reserveAndBuy(...)` — không biết đang dùng Naive hay OCC
- Chuyển strategy = thay `@Qualifier` hoặc config property — zero code change trong controller
- Test từng strategy độc lập — mock interface cho unit test

**Runtime switching:**
```yaml
ticketing:
  strategy: pessimistic  # naive | pessimistic | occ | serializable
```
→ Spring `@ConditionalOnProperty` hoặc `@Qualifier` inject đúng implementation

### 2. RetryWithBackoff — Exponential Backoff + Jitter

```
Attempt 1: fail → wait 50ms + jitter(0-25ms)
Attempt 2: fail → wait 100ms + jitter(0-50ms)
Attempt 3: fail → wait 200ms + jitter(0-100ms)
Attempt 4: fail → wait 400ms + jitter(0-200ms)
Attempt 5: fail → give up, throw exception
```

**Tại sao exponential?**
- Linear backoff: nếu 100 threads đều retry sau 50ms → thundering herd lại xảy ra
- Exponential: mỗi lần chờ gấp đôi → spread requests ra theo thời gian

**Tại sao jitter?**
- Không có jitter: 100 threads cùng fail ở T=0, cùng retry ở T=50ms → synchronized retry storm
- Có jitter: retry time = base_delay + random(0, base_delay/2) → desynchronize threads
- AWS recommends "Full Jitter" = random(0, base_delay) — simple và effective

**Config từ application.yml:**
```yaml
ticketing:
  retry:
    max-attempts: 5
    initial-delay-ms: 50
    max-delay-ms: 2000
    multiplier: 2.0
    jitter: true
```

### 3. ConflictMetrics — Micrometer Counters

| Metric | Khi nào tăng | Strategy |
|--------|-------------|----------|
| `ticketing.conflicts` | OCC version mismatch, serialization failure | 2.C, 2.D |
| `ticketing.retries` | Mỗi lần retry (bất kể thành/bại) | 2.C, 2.D |
| `ticketing.deadlocks` | PSQLException state 40P01 | 2.B |
| `ticketing.timeouts` | Lock wait timeout, connection timeout | 2.B |
| `ticketing.oversells` | available_seats < 0 detected (should be 0!) | All |
| `ticketing.successes` | Ticket created thành công | All |
| `ticketing.failures` | Request fail sau max retries | All |

→ Tất cả metrics có tag `strategy=naive|pessimistic|occ|...` để filter trong benchmark.

### 4. IdempotencyFilter — Duplicate Request Guard

```
Request đến với header: Idempotency-Key: abc-123

  → Check Redis: GET idempotency:abc-123
    → Nếu có: return cached response (200 OK + same body)
    → Nếu không: 
      → SET idempotency:abc-123 "PROCESSING" EX 3600
      → Forward request to controller
      → Controller trả response
      → SET idempotency:abc-123 {response_json} EX 3600
      → Return response
```

**Race condition trong filter:**
- 2 requests cùng key đến cùng lúc
- Redis SET NX (set if not exists) → chỉ 1 request thắng
- Request thua nhận PROCESSING → wait + retry hoặc return 409 Conflict

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Strategy Pattern = swap without controller change | Thêm 1 abstraction layer, over-engineering nếu chỉ có 1 strategy |
| Shared retry logic = consistent behavior | Config chung cho mọi strategy, nhưng optimal config khác nhau per strategy |
| Micrometer counters = quantitative comparison | Counter overhead (negligible nhưng có) |
| IdempotencyFilter = duplicate protection | Redis dependency, SPOF nếu Redis down |

## Khi nào dùng trong production

✅ Strategy Pattern: **luôn dùng** khi có nhiều implementation cho cùng 1 interface
✅ RetryWithBackoff: **luôn dùng** khi có transient failures (network, lock contention, serialization)
✅ ConflictMetrics: **luôn dùng** — observability là non-negotiable trong production
✅ IdempotencyFilter: **dùng cho mọi mutating endpoint** — network retry = duplicate risk

❌ IdempotencyFilter với in-memory cache thay Redis: mất idempotency khi app restart hoặc multi-instance

## Spring Boot Implementation Notes

### Strategy Pattern wiring
```java
@Configuration
public class StrategyConfig {
    @Bean
    @ConditionalOnProperty(name = "ticketing.strategy", havingValue = "pessimistic")
    public TicketingStrategy pessimistic(...) { return new PessimisticTicketingStrategy(...); }
}
```
Hoặc đơn giản hơn: tất cả strategies là `@Component`, inject bằng `Map<String, TicketingStrategy>`, chọn theo config.

### RetryWithBackoff — functional style
```java
retryWithBackoff.execute(() -> {
    // transactional code that might throw conflict exception
    return ticketRepository.reserveAndBuy(eventId, userId, seatLabel);
});
```
Nhận `Supplier<T>`, wrap trong retry loop, throw sau max attempts.

### IdempotencyFilter — servlet filter
```java
@Component
@Order(Ordered.HIGHEST_PRECEDENCE)
public class IdempotencyFilter extends OncePerRequestFilter { ... }
```
Filter chạy trước controller — transparent cho business logic.
