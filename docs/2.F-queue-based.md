# 2.F — Queue-based Per-Event

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích tại sao queue serialize writes ở application level — và tại sao đây là giải pháp "production-grade" cuối cùng
- Tôi có thể giải thích per-event queue — tại sao 1 queue per event thay vì 1 global queue
- Tôi có thể giải thích back-pressure — khi queue đầy, hệ thống phản hồi ra sao
- Tôi có thể giải thích tradeoff latency vs throughput vs fairness trong queue-based design
- Tôi có thể so sánh tất cả 6 strategies và biết khi nào dùng cái nào

## The Mental Model

Hình dung **quầy bán vé tại nhà ga**, mỗi tuyến xe (event) có **1 quầy riêng**:

1. Khách hàng đến → **xếp hàng** vào đúng quầy của tuyến xe mình cần
2. Nhân viên (worker) phục vụ **từng người một** (serialize) — không bao giờ 2 người cùng lúc
3. Nếu hàng quá dài → bảng hiện "Hàng đợi đầy, vui lòng quay lại sau" (back-pressure)
4. Mỗi quầy hoạt động **độc lập** — quầy tuyến A bận không ảnh hưởng quầy tuyến B

**Khác biệt cốt lõi:**
- Strategies A-D: serialize ở **DB level** (locks, versions, isolation)
- Strategy E: serialize ở **reserve step** (pessimistic lock ngắn)
- Strategy F: serialize ở **application level** (queue) — DB chỉ nhận 1 write tại 1 thời điểm per event

```
Strategies A-D:
  200 threads ──→ [DB] ← contention xảy ra TẠI ĐÂY
  
Strategy F:
  200 threads ──→ [QUEUE per event] ──→ [1 WORKER] ──→ [DB]
                  ← contention ở đây     ← serialize      ← zero contention
                  (Redis LPUSH O(1))      (1 thread)        (1 write at a time)
```

## Tại sao strategies trước chưa đủ

### Pessimistic (2.B) — lock contention ở DB
```
200 concurrent → 200 connections chờ lock → connection pool (20) exhausted
→ 180 threads chờ connection → timeout
→ DB lock manager overhead: memory + CPU cho mỗi waiter
```

### OCC (2.C) / SERIALIZABLE (2.D) — retry storm
```
200 concurrent → ~150 conflict → 150 retry → ~100 conflict lần 2 → ...
→ Wasted work: SELECT + business logic chạy rồi bỏ
→ DB load tăng gấp 3-5x so với actual successful writes
```

### Reservation (2.E) — vẫn lock DB khi reserve
```
Reserve step vẫn dùng pessimistic lock → contention vẫn ở DB level
Chỉ giảm THỜI GIAN contention, không giảm SỐ LƯỢNG contention points
```

### Queue — eliminates DB contention entirely
```
200 concurrent → 200 LPUSH vào Redis queue (O(1), ~0.1ms each)
→ 1 worker RPOP → process → 1 DB write (zero contention)
→ DB load = exactly 1 write per ticket (no wasted work)
→ Redis handles the serialization, not PostgreSQL
```

## Cơ chế hoạt động (DB/OS level)

### Redis List as Queue

```
Redis List: LPUSH (enqueue) + BRPOP (blocking dequeue)

Enqueue (producer — HTTP thread):
  LPUSH queue:event:1 "{eventId:1, userId:abc, seatLabel:A1}"
  → O(1), non-blocking, ~0.1ms
  → HTTP thread return ACCEPTED (202) ngay lập tức

Dequeue (consumer — worker thread):
  BRPOP queue:event:1 30
  → Blocking pop — chờ tối đa 30s nếu queue rỗng
  → Khi có item: pop + process + DB write
  → Worker xử lý tuần tự — 1 item tại 1 thời điểm
```

### Per-Event Queue — tại sao không 1 global queue

```
Global queue:
  Event 1: 100 requests ─┐
  Event 2: 100 requests ──┼──→ [1 GLOBAL QUEUE] ──→ [1 WORKER]
  Event 3: 100 requests ─┘
  → Worker xử lý 300 requests tuần tự
  → Event 2 phải chờ Event 1 xong — cross-event blocking!

Per-event queue:
  Event 1: 100 requests ──→ [QUEUE:event:1] ──→ [WORKER 1]
  Event 2: 100 requests ──→ [QUEUE:event:2] ──→ [WORKER 2]
  Event 3: 100 requests ──→ [QUEUE:event:3] ──→ [WORKER 3]
  → Mỗi event xử lý song song
  → Không cross-event blocking
  → Throughput scale linearly với số events
```

### Back-pressure — khi queue đầy

```
Queue max size = configurable (e.g., 10,000)

Khi LPUSH:
  Kiểm tra LLEN queue:event:1
  → Nếu < max_size: LPUSH thành công → return 202 ACCEPTED
  → Nếu >= max_size: reject → return 503 SERVICE_UNAVAILABLE

Client nhận 503 → hiển thị "Hệ thống đang bận, vui lòng thử lại"
→ Back-pressure truyền từ server → client → user
→ Hệ thống không bao giờ overload — graceful degradation
```

### Kết quả trả về — async notification

```
Vì queue-based là async, client không nhận kết quả ngay:

Option A — Polling:
  POST /tickets/reserve-and-buy → 202 {requestId: "abc"}
  GET  /tickets/status/abc → 200 {status: "PROCESSING"}
  GET  /tickets/status/abc → 200 {status: "SUCCESS", ticketId: 42}

Option B — CompletableFuture (trong project này):
  POST /tickets/reserve-and-buy
  → Enqueue + create CompletableFuture
  → Worker completes future khi xong
  → HTTP thread chờ future.get(timeout)
  → Return result synchronously
  → Nếu timeout → return 202 + polling
```

## SQL / Implementation

```java
// Producer (HTTP thread) — enqueue request
public TicketResult reserveAndBuy(Long eventId, UUID userId, String seatLabel) {
    // 1. Kiểm tra queue capacity (back-pressure)
    long queueSize = redis.opsForList().size("queue:event:" + eventId);
    if (queueSize >= maxQueueSize) return SERVICE_UNAVAILABLE;
    
    // 2. Tạo CompletableFuture cho response
    String requestId = UUID.randomUUID().toString();
    CompletableFuture<TicketResult> future = new CompletableFuture<>();
    pendingRequests.put(requestId, future);
    
    // 3. Enqueue (O(1))
    redis.opsForList().leftPush("queue:event:" + eventId,
        serialize(new QueueTicketRequest(requestId, eventId, userId, seatLabel)));
    
    // 4. Chờ worker xử lý (với timeout)
    return future.get(30, TimeUnit.SECONDS);
}

// Consumer (worker thread) — process 1 at a time
void processNext(Long eventId) {
    String json = redis.opsForList().rightPop("queue:event:" + eventId, 30, TimeUnit.SECONDS);
    QueueTicketRequest request = deserialize(json);
    
    // DB operation — ZERO contention vì chỉ 1 worker per event
    TicketResult result = doPurchase(request);  // pessimistic style, nhưng zero contention
    
    // Complete the future → HTTP thread nhận kết quả
    CompletableFuture<TicketResult> future = pendingRequests.remove(request.requestId());
    future.complete(result);
}
```

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| **Zero DB contention** — 1 write at a time | Infrastructure complexity — cần Redis |
| **Perfect fairness** — FIFO order | Latency thêm queue hop (~1ms) |
| **Back-pressure** — graceful degradation | Single point of failure — Redis down = no tickets |
| **Throughput predictable** — 1000/T per event | Worker failure → stuck requests (cần dead letter queue) |
| **No retry, no conflict, no deadlock** | Async complexity — CompletableFuture/polling |
| **Scale with events** — N events = N parallel workers | Memory: Redis queue size per event |
| **DB load = successful writes only** | Debugging harder — request → queue → worker → DB |

## Latency / Throughput Analysis

### Queue-based throughput
```
T = DB transaction time per ticket = 5ms
Throughput per event = 1/T = 200 TPS (1 worker)
Throughput per event = N/T = N × 200 TPS (N workers, partitioned seats)

200 concurrent, 1 event:
  Enqueue: 200 × 0.1ms = 20ms (parallel)
  Process: 200 × 5ms = 1000ms (serial)
  
  p50: 100 × 5ms = 500ms (chờ 100 requests trước)
  p95: 190 × 5ms = 950ms
  p99: 198 × 5ms = 990ms
```

### So sánh tất cả 6 strategies

```
200 concurrent, 100 seats, T=5ms:

Strategy     | Oversell | p50     | p95     | DB load
─────────────|──────────|─────────|─────────|────────
A Naive      | YES      | 5ms     | 10ms    | 200 writes (nhiều sai)
B Pessimistic| 0        | 500ms   | 950ms   | 200 writes (serialize)
C OCC        | 0        | 50ms    | 500ms   | ~400 writes (retries)
D Serializable| 0       | 60ms    | 600ms   | ~450 writes (retries+FP)
E Reservation| 0        | 500ms*  | 950ms*  | 200 writes (* chỉ reserve)
F Queue      | 0        | 500ms   | 950ms   | 200 writes (exact)

* Reservation: p50/p95 chỉ cho reserve step, confirm thêm ~5ms nhưng async
```

## Real-World Context

### Ticketmaster — Virtual Queue
- "Queue-it" integration: users enter virtual queue before accessing checkout
- Queue position displayed: "You are #4,521 in line"
- Server processes from queue head — exactly queue-based pattern
- Handles 1M+ concurrent for Taylor Swift concerts

### Discord — Message Queue per Channel
- Mỗi channel = 1 queue
- Messages serialize per channel (ordering guarantee)
- Cross-channel parallel — same pattern as per-event queue

### Kafka — Event Streaming at Scale
- Topic partitions = per-event queues
- 1 consumer per partition = serialize per partition
- Cross-partition parallel
- LinkedIn: 7 trillion messages/day

## Khi nào dùng trong production

✅ Phù hợp:
- **Extreme contention** — flash sale, 10K+ concurrent cho 1 event
- **Fairness required** — FIFO processing, không "lucky thread wins"
- **Predictable throughput** — capacity planning dễ dàng
- **Multi-instance deployment** — Redis queue shared across instances
- **Zero wasted work** — mỗi DB write = 1 successful ticket

❌ Không phù hợp:
- **Low contention** — overhead không đáng, dùng OCC
- **Low latency required** — queue hop thêm ~1ms + queue wait time
- **Simple deployment** — thêm Redis dependency
- **Real-time bidding** — cần instant response, queue quá chậm

→ Queue-based là "production-grade" pattern cho **flash sale / high contention**
→ Nếu contention thấp: OCC (2.C) đủ tốt và đơn giản hơn

### CompletableFuture for sync-over-async
```java
// ConcurrentHashMap<String, CompletableFuture<TicketResult>> pendingRequests
// Producer: pendingRequests.put(requestId, new CompletableFuture<>())
// Consumer: pendingRequests.remove(requestId).complete(result)
// HTTP thread: future.get(30, TimeUnit.SECONDS)
```

### Worker lifecycle
```java
@PostConstruct: start worker threads (1 per known event, or dynamic)
@PreDestroy: graceful shutdown — finish current item, drain queue
Dead letter: nếu worker crash → request stuck → timeout → client retry
```

### Gotcha: single-instance limitation
```
Trong project này: 1 worker per event per instance.
Production: cần distributed worker coordination:
  - Redis BRPOP đã atomic — nhiều instances BRPOP cùng queue → mỗi instance nhận 1 item
  - NHƯNG: nếu 2 instances cùng BRPOP → mất serialization guarantee!
  - Giải pháp: 1 consumer per queue (hoặc partitioned seats)
```
