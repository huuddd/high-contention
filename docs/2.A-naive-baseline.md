# 2.A — Naive Baseline (Intentionally Broken)

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích TOCTOU (Time-Of-Check to Time-Of-Use) race condition ở DB level
- Tôi có thể giải thích tại sao READ COMMITTED isolation KHÔNG ngăn oversell
- Tôi có thể giải thích PostgreSQL MVCC — mỗi transaction thấy snapshot riêng
- Tôi có thể vẽ timeline cụ thể cho thấy 2+ transactions cùng thấy available_seats=1 và cùng UPDATE thành công
- Tôi có thể giải thích tại sao "check rồi update" không bao giờ safe mà không có locking hoặc atomic operation

## The Mental Model

Hình dung 100 người xếp hàng mua vé, nhưng **không có quầy bán vé** — thay vào đó, mỗi người tự nhìn bảng thông báo "Còn 1 vé", tự quyết định mua, rồi tự sửa bảng.

Vấn đề: 50 người cùng nhìn bảng **cùng lúc**, cùng thấy "Còn 1 vé", cùng quyết định mua.
Kết quả: 50 người đều nghĩ mình mua được, nhưng thực tế chỉ có 1 vé.

Đây chính là **Naive approach** — đọc rồi ghi, không có cơ chế đồng bộ nào.

## Tại sao Naive bị sai — Race Condition Timeline

### Kịch bản: 1 event, available_seats = 1, 2 concurrent threads

```
T=0ms   Thread A: BEGIN
T=0ms   Thread B: BEGIN

T=1ms   Thread A: SELECT available_seats FROM events WHERE id = 1
                  → Kết quả: available_seats = 1  ✅ "Còn vé"
                  
T=1ms   Thread B: SELECT available_seats FROM events WHERE id = 1
                  → Kết quả: available_seats = 1  ✅ "Còn vé" (cùng snapshot!)

T=2ms   Thread A: UPDATE events SET available_seats = available_seats - 1 WHERE id = 1
                  → available_seats = 0  ✅
                  
T=2ms   Thread B: UPDATE events SET available_seats = available_seats - 1 WHERE id = 1
                  → available_seats = -1  ❌ OVERSELL!
                  (Nhưng CHECK constraint sẽ chặn nếu available_seats = 0 trước UPDATE)
                  
T=3ms   Thread A: INSERT INTO tickets (event_id, seat_id, user_id) VALUES (1, 42, 'user-A')
                  → ✅ Ticket tạo thành công

T=3ms   Thread B: INSERT INTO tickets (event_id, seat_id, user_id) VALUES (1, 42, 'user-B')
                  → ❌ UNIQUE violation (event_id, seat_id) — nhưng Thread B đã giảm counter!

T=4ms   Thread A: COMMIT ✅
T=4ms   Thread B: COMMIT hoặc ROLLBACK tùy exception handling
```

### Vấn đề sâu hơn — MVCC trong PostgreSQL

**READ COMMITTED** (default isolation level):
- Mỗi **statement** thấy snapshot mới nhất **tại thời điểm statement bắt đầu**
- Thread A và Thread B chạy SELECT gần như cùng lúc → cùng thấy snapshot với `available_seats = 1`
- UPDATE của Thread B thấy row đã bị Thread A modify → PostgreSQL **re-evaluates** WHERE clause
- Nhưng `available_seats - 1` vẫn được tính từ **giá trị hiện tại** (0), kết quả = -1

**Vậy CHECK constraint có cứu được không?**
- `CHECK (available_seats >= 0)` sẽ **reject** UPDATE nếu kết quả < 0
- NHƯNG: nếu 2 threads cùng SELECT thấy `available_seats = 2`, cùng UPDATE `-1`:
  - Thread A: 2 → 1 ✅
  - Thread B: 1 → 0 ✅ (cả hai thành công, bán 2 vé cho 2 người khác nhau)
  - Vấn đề: cả hai có thể INSERT ticket cho **cùng 1 seat**!
- CHECK constraint chỉ bảo vệ counter, **KHÔNG bảo vệ seat-level double-book**
- UNIQUE constraint trên `(event_id, seat_id)` là tuyến phòng thủ cuối → 1 trong 2 sẽ fail

### Kết luận: 2 lớp lỗi

1. **Counter-level oversell**: `available_seats` giảm xuống dưới thực tế → CHECK constraint chặn
2. **Seat-level double-book**: 2 users cùng INSERT ticket cho 1 seat → UNIQUE constraint chặn

→ DB constraints là **safety net**, nhưng Naive approach tạo ra **rất nhiều failed requests** và **inconsistent state** (counter lệch so với thực tế).

## Cơ chế hoạt động (DB/OS level)

### PostgreSQL MVCC — Multi-Version Concurrency Control

```
Mỗi row trong PostgreSQL có 2 hidden columns:
  xmin = transaction ID tạo row này
  xmax = transaction ID xóa/update row này (0 nếu chưa)

Khi UPDATE:
  1. PG KHÔNG sửa row cũ — tạo row MỚI (new tuple)
  2. Row cũ: xmax = current_txid
  3. Row mới: xmin = current_txid, xmax = 0
  4. Cả 2 row tồn tại cho đến khi VACUUM cleanup

Khi SELECT (READ COMMITTED):
  - Thấy row có xmin < current_snapshot AND (xmax = 0 OR xmax > current_snapshot)
  - Nghĩa là: thấy row đã committed TRƯỚC snapshot, chưa bị xóa/update
```

**Hệ quả cho Naive:**
- Thread A UPDATE → tạo tuple mới (available_seats = 0)
- Thread B SELECT → nếu A chưa COMMIT, B vẫn thấy tuple cũ (available_seats = 1)
- Thread B UPDATE → PostgreSQL phát hiện row đã bị modified, **chờ A COMMIT**
- Sau A COMMIT, B **re-evaluates** WHERE clause trên row mới → `available_seats - 1 = -1`
- CHECK constraint fires → B bị reject

Nhưng nếu `available_seats = 10` và 15 threads cùng chạy:
- Tất cả SELECT thấy 10 → tất cả nghĩ "còn vé"
- Chỉ 10 thành công, 5 fail tại CHECK hoặc UNIQUE
- **User experience tệ**: 5 users nghĩ mình mua được nhưng fail ở commit time

## SQL / Implementation

```java
// NAIVE — intentionally broken: check-then-act without any synchronization
public TicketResult reserveAndBuy(Long eventId, UUID userId, String seatLabel) {
    // Bước 1: SELECT (không lock) — tất cả threads cùng thấy snapshot
    Event event = eventRepository.findById(eventId);  // plain SELECT
    if (event.getAvailableSeats() <= 0) return SEAT_NOT_AVAILABLE;
    
    // ⚠️ TOCTOU GAP: giữa check và update, thread khác có thể đã mua
    
    Seat seat = seatRepository.findByEventIdAndSeatLabel(eventId, seatLabel);
    if (!seat.isAvailable()) return SEAT_NOT_AVAILABLE;
    
    // ⚠️ TOCTOU GAP: seat có thể đã bị thread khác đổi status
    
    // Bước 2: UPDATE counter (không version check)
    eventRepository.decrementAvailableSeats(eventId);
    
    // Bước 3: UPDATE seat status
    seatRepository.markAsSold(seat.getId());
    
    // Bước 4: INSERT ticket
    Ticket ticket = new Ticket(event, seat, userId, null);
    ticketRepository.save(ticket);
    
    return SUCCESS;
}
```

**3 TOCTOU gaps:**
1. Giữa `findById` và `decrementAvailableSeats` → counter có thể đã thay đổi
2. Giữa `findByEventIdAndSeatLabel` và `markAsSold` → seat có thể đã bị sold
3. Giữa `markAsSold` và `save(ticket)` → UNIQUE constraint có thể fail

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| Đơn giản nhất, dễ hiểu | **Oversell** dưới concurrent load |
| Latency thấp nhất (không lock, không retry) | **Double-book** nếu UNIQUE constraint không chặn |
| Throughput cao nhất (không blocking) | DB constraints = safety net, nhưng tạo inconsistent state |
| Zero overhead | Failed requests tăng tuyến tính với concurrency |

## Latency / Throughput Analysis

**Naive throughput (lý thuyết):**
- Không lock → không blocking → throughput = DB write throughput
- Single PostgreSQL: ~10,000-50,000 simple writes/sec
- Nhưng "throughput" này bao gồm cả **wrong results**

**Naive latency:**
- p50 ≈ 2-5ms (SELECT + UPDATE + INSERT, no waiting)
- p95 ≈ 5-10ms
- Rất thấp vì **không ai chờ ai**

**So sánh dự kiến với Pessimistic (2.B):**
- Naive p95 ≈ 5ms vs Pessimistic p95 ≈ 100-500ms
- Nhưng Naive oversell > 0 vs Pessimistic oversell = 0

## Khi nào dùng trong production

✅ Phù hợp: **KHÔNG BAO GIỜ** cho write-contention scenarios

✅ Phù hợp (biến thể): Read-heavy với eventual consistency chấp nhận được
  - Ví dụ: "like" counter, view counter — oversell không phải vấn đề

❌ Không phù hợp: **Mọi bối cảnh có scarce resources** — ticketing, inventory, booking

→ Naive tồn tại trong project này chỉ để **chứng minh vấn đề bằng số liệu benchmark**

## Spring Boot Implementation Notes

### Transaction trong Naive
```java
@Transactional  // vẫn cần — nhưng chỉ đảm bảo atomicity, KHÔNG đảm bảo isolation
```
- `@Transactional` với default isolation (READ_COMMITTED) **KHÔNG ngăn** TOCTOU
- Mỗi statement trong transaction thấy snapshot mới, nhưng check-then-act vẫn bị race

### Tại sao vẫn cần @Transactional?
- Atomicity: nếu INSERT ticket fail (UNIQUE violation), ROLLBACK cả UPDATE counter
- Không có @Transactional: counter bị decrement nhưng ticket không tạo → **counter drift**
- Counter drift: `available_seats` không khớp với thực tế → mọi strategy sau đều sai

### Exception handling
- Catch `DataIntegrityViolationException` cho UNIQUE violation
- Catch `ConstraintViolationException` cho CHECK constraint violation
- Log và count vào ConflictMetrics — benchmark cần số liệu
