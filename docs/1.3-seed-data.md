# 1.3 — Seed Data

## Mục tiêu học tập

Sau task này bạn hiểu được:
- Tôi có thể giải thích tại sao seed data cần idempotent — chạy nhiều lần không tạo duplicate
- Tôi có thể giải thích tại sao test data phải deterministic — cùng seed = cùng kết quả
- Tôi có thể giải thích mối quan hệ giữa `events.available_seats` và count thực tế của `seats` rows

## The Mental Model

Seed data giống như **chuẩn bị phòng hòa nhạc trước khi mở bán vé**:
- Tạo 1 buổi hòa nhạc (event) — "Concert Alpha"
- Đặt 100 ghế vào phòng (seats A1–A100) — tất cả đều AVAILABLE
- `available_seats = 100` phải **khớp chính xác** với số rows `seats` có status = 'AVAILABLE'

Nếu 2 con số này lệch nhau → mọi concurrency strategy sẽ cho kết quả sai.

## Tại sao seed data quan trọng cho benchmark

### Consistency giữa `events.available_seats` và `seats` count

```
events.available_seats = 100
COUNT(seats WHERE status = 'AVAILABLE') = 100
```

Nếu lệch:
- `available_seats = 100` nhưng chỉ có 50 seat rows → strategy sẽ decrement counter xuống 0
  nhưng thực tế chỉ có 50 ticket có thể tạo → **phantom availability**
- `available_seats = 50` nhưng có 100 seat rows → strategy dừng bán ở 50
  nhưng thực tế còn 50 ghế trống → **lost revenue**

→ Seed data phải đảm bảo **2 nguồn truth nhất quán** từ đầu.

### Idempotent seed — tại sao cần?

Developer sẽ chạy `make seed` nhiều lần:
- Lần 1: tạo data
- Lần 2 (sau khi test): cần reset về trạng thái ban đầu
- Lần N: benchmark cần fresh data mỗi lần chạy

→ Seed script phải: **TRUNCATE → INSERT** (hoặc DELETE + INSERT)
→ Không dùng `INSERT ... ON CONFLICT DO NOTHING` vì không reset state cũ

### Deterministic data

- Seat labels: A1, A2, ..., A100 — không random
- Event ID: dùng explicit ID hoặc RETURNING để biết chính xác
- Mọi lần chạy seed → cùng trạng thái → benchmark reproducible

## SQL Implementation

```sql
-- Truncate với CASCADE để xóa dependent rows (tickets, reservations)
TRUNCATE TABLE reservations, tickets, seats, events RESTART IDENTITY CASCADE;

-- 1 event: Concert Alpha, 100 ghế
INSERT INTO events (name, total_seats, available_seats, version)
VALUES ('Concert Alpha', 100, 100, 0);

-- 100 seats: A1 → A100, tất cả AVAILABLE
INSERT INTO seats (event_id, seat_label, status)
SELECT 1, 'A' || gs, 'AVAILABLE'
FROM generate_series(1, 100) AS gs;
```

**Giải thích:**
- `TRUNCATE ... RESTART IDENTITY CASCADE` — reset auto-increment, xóa tất cả dependent rows
- `generate_series(1, 100)` — PostgreSQL function, tạo 100 rows trong 1 statement
- Event ID = 1 (vì RESTART IDENTITY reset BIGSERIAL về 1)

## Trade-offs

| Ưu điểm | Nhược điểm |
|---------|-----------|
| TRUNCATE + INSERT = idempotent, clean state | TRUNCATE = DDL-like, acquire AccessExclusiveLock |
| generate_series = 1 statement cho 100 rows | Chỉ phù hợp PostgreSQL, không portable |
| Deterministic IDs (event=1, seats=1-100) | Hard-coded assumptions trong test scripts |
| Simple, readable | Chỉ 1 event — chưa test multi-event scenarios |

## Khi nào dùng trong production

✅ Phù hợp:
- Local development — quick reset giữa các test runs
- CI/CD pipeline — seed trước integration test
- Demo environment — consistent demo data

❌ Không phù hợp:
- Production — không bao giờ TRUNCATE production tables
- Staging với real-ish data — dùng anonymized production dump thay vì synthetic seed

## Benchmark scenarios sử dụng seed data

| Scenario | Setup cần | Seed đáp ứng? |
|----------|-----------|---------------|
| B1: Stock=1 | 1 event, 1 seat | Cần seed riêng hoặc chỉ dùng seat A1 |
| B2: Hot-seat | 1 event, seat "A15" | ✅ A15 có trong 100 seats |
| B3: Burst | 1 event, 100 seats | ✅ Chính xác |

→ B1 (Stock=1) sẽ cần modify seed hoặc tạo event riêng với 1 seat. K6 script sẽ handle.
